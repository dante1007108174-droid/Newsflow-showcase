## 1. 背景与问题定义

NewsFlow.ai 的核心链路可以拆成两类任务：

1. **调度类任务 (Router / Orchestration)**
   - 目标：识别用户意图、提取槽位、决定是否/调用哪个工作流、把参数填对。
   - 特征：输入短、要求稳定、错误成本高（一旦填错参数就会"做错事"）。

2. **生成类任务 (Generation / Writing)**
   - 目标：面向用户输出高质量摘要、结构化简报、邮件 HTML。
   - 特征：上下文长、输出长、质量影响体验上限，成本主要来自输出 token。

因此模型选择不能只看"聪明不聪明"，而要满足两个目标同时成立：

- **能力不溢出**：该用快模型的地方不要用大模型，避免成本浪费、延迟增加。
- **能力要够用**：该用强模型的地方要用强模型，避免摘要质量或逻辑硬伤。

结论不是"选一个最强模型"，而是"按节点分层，选最合适模型"。

---

## 2. 候选模型与约束

本轮实测对比的候选模型：

- **Doubao-Seed-1.6**：Coze 平台原生模型，接入成本低、响应快。
- **DeepSeek-V3.1（用于调度评测）/ DeepSeek-V3.2（用于生成落地）**：逻辑与生成能力强，输出成本更可控。

说明：

- 其他模型（如 Kimi/GLM/Qwen 等）本阶段未纳入实测，原因是项目目标优先级为"先跑通可用产品"，选择了接入最顺滑且性价比明确的组合。
- 调度层评测聚焦"意图识别 + 槽位填充 + 指令遵循"，不是比作文能力。

## 3. 评测体系设计

### 3.1 评测对象

评测对象限定为 **Model 1：总控-调度**（Coze Agent 的路由/调度能力），不包含下游 RSS 抓取、Top-K 预筛选、邮件发送等工程耗时。

评测配置：temperature = **0.5**（降低随机性对槽位遵循的影响）。

### 3.2 评测维度与权重

| 维度 | 权重 | 评估问题 | 典型失败后果 |
| --- | --- | --- | --- |
| 意图精准度 | 30% | "用户想做什么"是否识别正确 | 调错工作流/走错流程 |
| 槽位/指令遵循 | 40% | 参数是否填对、是否遵循格式/约束 | 邮箱/关键词/来源等被填错 |
| 内容质量 | 20% | 在无需工具时回复是否自然且可用 | 追问不清晰、话术失控 |
| 响应速度 | 10% | 首次响应是否足够快 | 交互卡顿、用户流失 |

### 3.3 测试集设计

共 18 个测试 Case，覆盖高频链路与边界场景：

- 即时查询：基础关键词、缺失关键词追问、频道查询、指定信源过滤、数量限制、跨语言、排除式查询
- 配置订阅：一次性完整配置、关键词/邮箱槽位补齐、模糊时间追问
- 推送执行：已配置发送、未配置引导、复合指令（更新邮箱 + 发送）
- 兜底/风控：Prompt 注入攻击、逻辑陷阱

## 4. 评测结果与洞察

### 4.1 维度平均分（调度层）

| 模型 | 意图精准度(30%) | 槽位/指令遵循(40%) | 内容质量(20%) | 响应速度(10%) |
| --- | ---:| ---:| ---:| ---:|
| DeepSeek-V3.1 | 10.00 | 9.17 | 9.30 | 8.56 |
| Doubao-Seed-1.6 | 10.00 | 10.00 | 8.75 | 8.61 |

关键解读：

- **意图识别**：两者都达到满分，"看懂用户要干什么"不是分歧点。
- **槽位/指令遵循**：豆包在 18/18 Case 保持满分，DeepSeek 在 3 个 Case 明显丢分（从 10 跌到 5）。调度层最怕"偶发不稳定"。
- **响应速度**：两者接近，差异不是决策关键。

### 4.2 典型差异 Case

DeepSeek 在调度层的问题不是"看不懂"，而是"有时会多做一步或夹带信息"：

| Case | 场景类型 | DeepSeek 问题 | 豆包表现 |
| --- | --- | --- | --- |
| Case 04 | 泛化参数提取 | 参数提取不够聚焦，附带其他信息 | 槽位干净，填参稳定 |
| Case 07 | 指定信源过滤 | source/key 组合提取不稳定 | 组合槽位稳定 |
| Case 13 | 跨语言指令 | language 等指令槽位遵循不稳定 | 指令遵循更稳定 |

调度层的本质是"确定性控制"：宁可"少说一点、但参数一定对"，也不要"很聪明、但偶尔自作主张"。

### 4.3 价格评分与成本结构

| 项目 | Doubao-Seed-1.6 | DeepSeek-V3 |
| --- | ---:| ---:|
| 输入（缓存未命中） | 0.8 元/万 token | 2 元/万 token |
| 输入（缓存命中） | 0.16 元/万 token | 0.2 元/万 token |
| 输出 | 8 元/万 token | 3 元/万 token |

- 调度层：输入短、输出短，豆包的输入价格与平台接入便利性更占优
- 生成层：输出长，DeepSeek 的输出价格更占优

## 5. 最终决策：分层模型策略

### 5.1 决策结论

采用"快模型做控制、强模型做生成"的分层架构：

- **调度/扩写/轻量推理**：`豆包-1.6-极致速度`
  - 典型节点：意图识别、槽位提取、关键词扩写
- **深度摘要/长文生成**：`DeepSeek-V3.2`
  - 典型节点：Top-K 新闻摘要、日报正文组织、邮件 HTML 渲染

### 5.2 代码节点与模型节点分工（v1.1 新增）

除了模型分层，本项目还在工作流内部做了"代码节点 vs 模型节点"的分工：

> 核心原则：确定性的事不交给 LLM，需要判断力的事不用硬编码。

| 任务 | 处理方式 | 理由 |
| --- | --- | --- |
| RSS 抓取与 XML 解析 | 代码节点（Python） | 纯确定性操作，模型做不了也不该做 |
| 时间解析与时区转换 | 代码节点（Python） | 多格式兼容需要精确逻辑，模型容易出错（Bug-03 的教训） |
| 预筛选（Top-K、去重、时效过滤） | 代码节点（Python） | 减少喂给模型的 token 量，降低成本和延迟（Bug-05 的教训） |
| 关键词扩写 | 模型节点（豆包 1.6） | 需要语义理解，但任务简单，轻量模型即可 |
| 新闻聚类、排序、摘要 | 模型节点（DeepSeek V3.2） | 需要深度理解和生成能力 |
| HTML 日报生成 | 模型节点（DeepSeek V3.2） | 需要结构化输出和排版能力 |

这个分工的效果：
- 响应时间从 >40s 降到 ~8s（代码节点预筛选减少了模型输入 token）
- 时间幻觉问题消除（代码节点强制北京时间，不依赖模型判断）
- 成本可控（只把真正需要理解力的任务交给模型）

### 5.3 成本估算方法

单次调用成本：`Cost = Tin/10000 * Pin + Tout/10000 * Pout`

工程层面的成本控制手段：
1. **Top-K 预筛选**：先用代码把候选新闻压缩到 20 条以内，再交给生成模型
2. **缓存复用**：定时生成 HTML 缓存，批量发送时复用同一份内容，避免重复生成

## 6. 风险与后续优化

### 6.1 已识别风险

- **豆包 1.6 即将下架**（v1.1 新增）：预计约一个月后下架，需提前测试替代轻量模型。替代标准：响应速度 ≤ 2s，关键词扩写和槽位遵循质量不低于当前水平。
- **模型行为漂移**：云端模型版本升级可能导致路由行为改变。
- **评测覆盖不足**：当前只比较了两种模型，Case 数量有限。

### 6.2 对策

- 将 18 Case 作为回归集，每次调整前后都跑一遍
- 豆包下架前完成替代模型评测和切换
- 后续若扩展候选模型，优先在调度层引入第三方候选做同口径对比

---

## 7. 结论（面试 30 秒回答模板）

我没有选"一个最强模型"，而是按业务节点分层：调度层需要稳定的意图识别和槽位填充，所以我用豆包 1.6；生成层需要高质量摘要和长输出，同时输出成本要可控，所以我用 DeepSeek V3.2。这个决策来自一套 18 个 Case 的评测体系，结果显示两者在意图识别都很强，但 DeepSeek 在调度层有 3 个场景槽位不稳定，而豆包保持全满分。同时在工作流内部，我把确定性任务（RSS 解析、时间转换、预筛选）放在代码节点，只把需要理解力的任务交给模型，这样响应时间从 40 多秒降到了 8 秒。
