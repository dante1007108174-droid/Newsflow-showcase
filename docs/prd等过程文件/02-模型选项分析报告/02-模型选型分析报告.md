# NewsFlow.ai 模型选型分析报告

> 文档版本: v1.0
> 
> 最后更新: 2026-02-03
> 
> 覆盖范围: Agent 调度层模型选型 + 内容生成层分层策略

---

## 1. 背景与问题定义

NewsFlow.ai 的核心链路可以拆成两类任务：

1. **调度类任务 (Router / Orchestration)**
   - 目标: 识别用户意图、提取槽位、决定是否/调用哪个工作流、把参数填对。
   - 特征: 输入短、要求稳定、错误成本高(一旦填错参数就会“做错事”)。

2. **生成类任务 (Generation / Writing)**
   - 目标: 面向用户输出高质量摘要、结构化简报、邮件 HTML。
   - 特征: 上下文长、输出长、质量影响体验上限，成本主要来自输出 token。

因此模型选择不能只看“聪明不聪明”，而要满足两个目标同时成立：

- **能力不溢出**: 该用快模型的地方不要用大模型，避免成本浪费、延迟增加。
- **能力不不足**: 该用强模型的地方要用强模型，避免摘要质量或逻辑硬伤。

结论不是“选一个最强模型”，而是“按节点分层，选最合适模型”。

---

## 2. 候选模型与约束

本轮实测对比的候选模型:

- **Doubao-Seed-1.6 (豆包 1.6 极致速度)**: Coze 平台原生模型，接入成本低、响应快。
- **DeepSeek-V3 (用于调度评测) / DeepSeek-V3.2 (用于生成落地)**: 逻辑与生成能力强，输出成本更可控。

说明:

- 其他模型(如 Kimi/GLM/Qwen 等)本阶段未纳入实测，原因是项目目标优先级为“先跑通可用产品”，选择了接入最顺滑且性价比明确的组合(豆包 + DeepSeek)。
- 调度层评测聚焦“意图识别 + 槽位填充 + 指令遵循”，不是比作文能力。

---

## 3. 评测体系设计

### 3.1 评测对象

评测对象限定为 **Model 1: 总控-调度**(Coze Agent 的路由/调度能力)，不包含下游 RSS 抓取、Top-K 预筛选、邮件发送等工程耗时。

评测配置说明:

- 调度层对比评测时，将模型 temperature 控制在 **0.5**，以降低随机性对“槽位/指令遵循”的影响。

评测数据来源:

- `docs/temp_sheet_9.json` (DeepSeek-V3 vs Doubao-Seed-1.6 对比，18 个 Case)
- `docs/temp_sheet_5.json` (核心质量指标摘要)

### 3.2 评测维度与权重

调度层质量维度定义如下:

| 维度 | 权重 | 评估问题 | 典型失败后果 |
| --- | --- | --- | --- |
| 意图精准度 | 30% | “用户想做什么”是否识别正确 | 调错工作流/走错流程 |
| 槽位/指令遵循 | 40% | 参数是否填对、是否遵循格式/约束 | 邮箱/关键词/来源等被填错 |
| 内容质量 | 20% | 在无需工具时回复是否自然且可用 | 追问不清晰、话术失控 |
| 响应速度 | 10% | 首次响应是否足够快 | 交互卡顿、用户流失 |

另外单独对 **价格维度(权重 10%)** 做了评分，作为“落地可持续性”约束(见第 4.4 节)。

### 3.3 测试集设计

共 18 个测试 Case，覆盖高频链路与边界场景:

- 即时查询: 基础关键词、缺失关键词追问、频道查询、指定信源过滤、数量限制、跨语言、排除式查询
- 配置订阅: 一次性完整配置、关键词/邮箱槽位补齐、模糊时间追问
- 推送执行: 已配置发送、未配置引导、复合指令(更新邮箱 + 发送)
- 兜底/风控: Prompt 注入攻击、逻辑陷阱

## 4. 评测结果与洞察

### 4.1 维度平均分(调度层)

基于 18 个 Case 的平均分(0-10 分制)如下(注: 内容质量并非每个 Case 都评分，详见 4.2):

| 模型 | 意图精准度(30%) | 槽位/指令遵循(40%) | 内容质量(20%) | 响应速度(10%) |
| --- | ---:| ---:| ---:| ---:|
| DeepSeek-V3 | 10.00 | 9.17 | 9.30 | 8.56 |
| Doubao-Seed-1.6 | 10.00 | 10.00 | 8.75 | 8.61 |

关键解读:

- **意图识别**: 两者在该测试集里都达到满分，说明“看懂用户要干什么”不是主要分歧点。
- **槽位/指令遵循**: 豆包在 18/18 Case 保持满分，DeepSeek 在 3 个 Case 明显丢分(从 10 跌到 5)。调度层最怕的就是这一类“偶发不稳定”。
- **响应速度**: 两者在首次响应速度评分上接近，差异不是决策关键。

### 4.2 内容质量维度的说明(避免误读)

在调度层评测中，很多 Case 的“预期结果”是“调用某个工具/工作流 + 参数正确”。

这类场景里，模型的主要职责是 **决策与填参**，并不承担长文本生成；因此“内容质量”维度在部分 Case 被标记为不评分(在原始表格里显示为 "-" )。

因此，内容质量平均分仅作为补充信息，**不应被当作路由模型的核心指标**。

### 4.3 典型差异 Case(为什么 DeepSeek 在调度层吃亏)

DeepSeek 在调度层的主要问题不是“看不懂”，而是“有时会多做一步或夹带信息”，导致槽位/参数不够干净。

在本次数据中，DeepSeek 明显丢分集中在以下类型:

| Case | 场景类型 | 用户输入(脱敏/概括) | DeepSeek 问题表现 | 豆包表现 |
| --- | --- | --- | --- | --- |
| Case 04 | 泛化参数提取 | “某公司股价怎么样” | 参数提取不够聚焦，可能附带其他信息 | 槽位干净，填参稳定 |
| Case 07 | 指定信源过滤 | “只看某信源关于某主题” | source/key 组合提取不稳定 | 组合槽位稳定 |
| Case 13 | 跨语言指令 | “用英语总结科技热点” | language 等指令槽位遵循不稳定 | 指令遵循更稳定 |

调度层的本质是“确定性控制”，对模型的偏好是:

- 宁可“少说一点、但参数一定对”
- 也不要“很聪明、但偶尔自作主张”

这就是最终选择豆包作为调度层模型的核心原因。

### 4.4 价格评分(权重 10%)与成本结构结论

价格评分表(来自 `docs/temp_sheet_9.json`)给出了一个对产品落地很实用的结论：

- **调度层**: 输入短、输出短，成本主要由输入与少量输出构成，豆包的输入价格与平台接入便利性更占优。
- **生成层**: 输出长(摘要/邮件 HTML)，成本主要由输出 token 决定，DeepSeek 的输出价格与计费可预测性更占优。

价格对比(按“每 1 万 token”口径，原始数据):

| 项目 | Doubao-Seed-1.6 | DeepSeek-V3 |
| --- | ---:| ---:|
| 输入(缓存未命中) | 0.8 元 / 万 token | 2 元 / 万 token |
| 输入(缓存命中) | 0.16 元 / 万 token | 0.2 元 / 万 token |
| 输出 | 8 元 / 万 token | 3 元 / 万 token |

价格评分结论(原始表):

- Doubao 价格综合评分: **7.1**
- DeepSeek 价格综合评分: **8.6**

注意: 这不与“调度层选豆包”矛盾，因为调度层更看重稳定与接入便利；而生成层才是输出成本的大头，DeepSeek 更适合放在生成层承担长输出。

## 5. 最终决策: 分层模型策略

### 5.1 决策结论

采用“快模型做控制、强模型做生成”的分层架构:

- **调度/扩写/轻量推理**: `豆包-1.6-极致速度`
  - 典型节点: 意图识别、槽位提取、是否调用工作流、关键词扩写
- **深度摘要/长文生成**: `DeepSeek-V3.2`
  - 典型节点: Top-K 新闻摘要、日报正文组织、邮件 HTML 渲染文本

该结论已在 PRD v2.0 的“模型分层策略”中固化(见 `docs/01-PRD-产品需求文档-v2.0.md`)。

### 5.2 为什么分层策略能同时提升体验与成本

从成本结构上看:

- 调度层的 token 规模小，但调用频繁，首响延迟敏感。
- 生成层的 token 规模大，但调用频次更可控(定时任务、生成缓存、按需触发)。

从质量结构上看:

- 调度层最重要的是“稳定地做对事”(工具调用与参数正确)。
- 生成层最重要的是“内容上限”(摘要质量、结构、可读性)。

分层的本质是把两个优化目标从同一个模型中解耦。

### 5.3 成本估算方法(可复用)

为了避免“靠感觉算账”，建议在后续版本使用统一口径估算成本:

设:

- 输入 token 数: Tin
- 输出 token 数: Tout
- 输入单价(元/万 token): Pin
- 输出单价(元/万 token): Pout

则单次调用成本约为:

`Cost = Tin/10000 * Pin + Tout/10000 * Pout`

在工程上，通过两种方式控制 Tout:

1. **Top-K 预筛选**: 先用代码把候选新闻压缩到 20 条以内，再交给生成模型总结。
2. **缓存复用**: 定时生成 HTML 缓存，批量发送时复用同一份内容，避免重复生成。

## 6. 风险与后续优化

### 6.1 已识别风险

- **模型行为漂移**: 云端模型版本升级可能导致路由行为改变(尤其是槽位遵循)。
- **评测覆盖不足**: 当前只比较了两种模型，且 Case 数量仍有限。

### 6.2 对策

- 将 18 Case 作为“回归集”，每次调整 Agent/System Prompt 或切换模型前后都跑一遍。
- 按月/按版本复评并记录趋势，避免“体验悄悄变差”。
- 后续若要扩展候选模型，优先在调度层引入第三方候选(例如 Qwen/GLM/Kimi)做同口径对比，避免凭主观印象决策。

---

## 7. 结论(面试 30 秒回答模板)

我没有选“一个最强模型”，而是按业务节点分层：调度层需要稳定的意图识别和槽位填充，所以我用豆包 1.6；生成层需要高质量摘要和长输出，同时输出成本要可控，所以我用 DeepSeek V3.2。这个决策来自一套 18 个 Case 的评测体系，结果显示两者在意图识别都很强，但 DeepSeek 在调度层有 3 个场景槽位不稳定，而豆包保持全满分；同时价格结构上输出成本是大头，因此把 DeepSeek 放到生成层能把成本和质量一起打到最优。
