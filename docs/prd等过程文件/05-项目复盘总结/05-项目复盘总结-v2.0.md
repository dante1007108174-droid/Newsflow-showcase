## 1. 项目定位与目标

NewsFlow.ai 是一个 AI 驱动的新闻聚合与订阅推送助手，核心能力是：

- 用户通过自然语言完成**即时查询**、**订阅管理**、**测试推送**
- 系统每天定时生成日报并通过邮件推送
- 通过评测体系把"模型能力"转成可控的"产品稳定性"

这个项目对我而言的目标不是做一个 Demo，而是完整跑通：

- 产品定义（PRD）→ 从 v1.0 迭代到 v3.0
- Prompt / 工作流设计 → 从 v0 演进到 v3（373 行模块化提示词）
- 模型选型与分层 → 18 Case 评测驱动的分层决策
- 前后端落地与迭代 → 含埋点、反馈、限流等基础设施
- 评测与复盘文档沉淀 → 24 个 Bad Case 的系统性修复

## 2. 时间线（五阶段）

整体周期约 3 个月，从去年 11 月中旬开始：

### Phase 1：原型验证（约 2 周）

- 搭建最小可用的 Coze Agent 与工作流，验证"能查到新闻 + 能跑通订阅/发送"的闭环
- 建立 Prompt 记录表（早期有记录，后续随任务增多记录减少）
- 主要产出：可用的最小交互链路 + 初版 Prompt

### Phase 2：评测体系与模型选型（约 1 个月，集中在 12 月）

- 补齐评测数据集与 Bad Case 库，明确什么叫"稳定"
- 做调度层模型选型评测（18 Case），输出模型分层决策
- 输出/完善 PRD（逐步升级为 v2.0）

### Phase 3：产品落地（集中在 1 月）

- 落地前端与后端（静态前端 + Node/Express API）
- 前后端落地后，对 Workflow 做工程化升级以适配真实调用链路
- 对 Agent 系统提示词做两版关键结构调整（v1→v2：user_id 注入；v2→v3：调用守卫 + 实体透传）

### Phase 4：工程完善与文档沉淀（1 月底 - 2 月初）

- 测试推送从即时生成切换为缓存方案（`send_mail_cache_using`）

- 补齐埋点系统 MVP
- 首页直接订阅路径（不依赖 AI 对话）
- 文档沉淀：PRD v2.0、模型选型报告、Prompt 工程手册、评测报告、项目复盘

### Phase 5：种子测试准备（当前，2 月中旬）

- PRD 从 v2.0 全面梳理升级到 v3.0
- 信源管理文档化
- 所有过程文件同步更新
- 准备种子用户测试（AI 主题优先）

## 3. 核心设计决策

### 3.1 分层模型策略

项目最重要的决策：不追求"一个模型打天下"，而是按节点分层：

- 调度/扩写/轻量推理：豆包 1.6（稳定的槽位遵循 + 接入便利）
- 深度摘要/长输出生成：DeepSeek V3.2（更强的理解力 + 输出成本可控）

这条决策来自 18 Case 的评测数据，不是拍脑袋。

### 3.2 代码节点与模型节点分工

确定性的事不交给 LLM，需要判断力的事不用硬编码：

- 代码节点：RSS 抓取、时间解析、预筛选（Top-K）
- 模型节点：关键词扩写、聚类排序、摘要生成

效果：响应时间从 >40s 降到 ~8s，时间幻觉问题消除。

### 3.3 缓存架构

生成与发送解耦，一次生成多次分发：

- 每日 08:00 生成日报 HTML → 写入 `daily_news_history`
- 定时推送和测试推送都从同一缓存读取
- 节省了重复生成的成本，保证了内容一致性

### 3.4 Prompt 工程的防御性设计

从 Bug 驱动中总结出的关键设计：

- 调用守卫（Calling Guard）：调用工作流前强制检查参数
- raw_query 透传：减少 Agent 层参数解析的出错机会
- 来源过滤自检：模型输出前逐条检查来源匹配
- keyword 实体透传：Agent 只做路由，细节交给下游

## 4. 做得好的部分

### 4.1 先建立评测，再谈优化

在模型应用里，很多体验问题是"看起来像模型问题，实际上是流程/约束问题"。

尽早建立评测集与 Bad Case 库：
- 让问题可复现、可讨论、可回归
- 让"Prompt 改动"从拍脑袋变成可验证
- 累计修复 24 个 Bad Case，形成了系统性的问题定位方法

### 4.2 关键约束前置（隐私 + 注入防御）

从 v2 版本开始就把安全约束放在最高优先级：
- user_id 自动提取，严禁向用户索要/泄露
- Prompt 注入防御
- 接口限流、敏感字段拦截

### 4.3 把 token/延迟当成一等公民

通过代码节点预筛选 + 缓存复用，把成本和延迟控制在合理范围。这不是事后优化，而是架构设计时就考虑的。

### 4.4 产品分层意识

从"只有 AI 对话"到"首页直接订阅 + AI 浮窗 + 定时推送"三条路径并存，每条路径服务不同的用户意图（获客、探索、留存）。

## 5. 做得不够好的部分（以及为什么）

### 5.1 记录体系中断

早期有 Prompt 迭代记录表，但随着工作量暴涨记录变少。复盘后认为至少应保留：
- "重大版本"变更点（为什么改、改了什么、风险是什么）
- "回归集"结果（改动前后是否退化）

### 5.2 顺序上有来回

最合理的顺序是：先定义边界（PRD）→ 再做评测 → 再做工程落地。但实际执行中为了快速验证可行性，前期更偏工程驱动，后期再补齐产品与评测框架。

### 5.3 信源管理缺乏文档化

信源配置散落在各个工作流的代码节点里，直到 v3.0 梳理时才发现即时查询和日报生成用的信源完全不同，且即时查询缺少财经和 AI 专业源。

---

## 6. 如果重来一次

1. **更早固化回归流程**：每次关键改动后都跑回归集，形成节奏
2. **Prompt 版本化**：重大版本必须有 changelog
3. **更早做信息结构化**：让上游数据尽量结构化后再给 LLM
4. **信源管理前置**：一开始就建立信源清单和分类，而不是散落在代码里
5. **缓存意识前置**：缓存不只是性能优化，更是架构设计——生成与分发解耦从一开始就该考虑
