## 1. 评测目标与范围

本报告回答三个问题：

1. **能不能用**：核心路径是否闭环，用户指令是否能被正确完成。
2. **稳不稳定**：多轮对话/边界场景下，意图与槽位是否保持一致。
3. **能不能迭代**：是否形成可复用的评测资产（回归集 + Bad Case 库），避免"改一次坏一次"。

评测范围覆盖：

- **调度层（Agent Router）**：意图识别、槽位提取、工作流选择与参数传递。
- **核心端到端链路（MVP）**：即时查询、订阅管理、推送触发等关键用户路径。
- **即时查询内容质量**：来源过滤、排序稳定性、输出格式一致性。

不覆盖/弱覆盖：

- 压测与并发容量
- 线上监控告警体系
- 真实用户规模与增长指标（种子测试阶段将开始收集）

## 3. 方法论：如何设计 Case

本项目把 Case 设计成"覆盖用户路径 + 覆盖失败模式"两条轴：

**用户路径覆盖**：
- 即时查询：主题缺失追问、信源过滤、数量限制、跨语言、来源别名
- 配置订阅：一次性给全信息、多轮补槽位、取消后重新订阅
- 推送执行：已配置发送、未配置引导、复合指令
- 兜底风控：Prompt 注入、无关话题

**失败模式覆盖**：
- 意图混淆（关键词输入被当成配置）
- 槽位丢失/覆盖（多轮中忘记上轮信息）
- 参数污染（关键词带冗余后缀）
- 来源过滤失效（指定来源但输出了其他来源的新闻）
- 安全边界缺失（泄露系统提示词/隐私字段）
- 调用守卫缺失（缺参数直接调用工作流）

### 4.4 Bad Case 共同规律

这些案例的共同点：

- 不是"模型不够聪明"，而是"约束不够明确/流程缺少校验"
- 修复往往不是加长 Prompt，而是把规则、校验与工程预处理放到合适的层
- 代码节点处理确定性任务（时间解析、预筛选），模型节点处理需要判断力的任务（聚类、排序）

## 5. 回归策略

### 5.1 当前回归策略

| 回归集 | 触发条件 | 验收标准 |
| --- | --- | --- |
| 路由回归（18 Case） | 调整 System Prompt、切换调度模型、调整意图表/槽位规则 | 槽位/指令遵循不退化 |
| MVP 回归（22 Case） | 工作流修改、前后端接口变更、数据结构变更 | 任务完成率保持 100% |
| Bug Bank 回归 | 每新增一个 Bad Case | 进入长期用例库 |

### 5.2 后续工程化方向

- 将回归用例脚本化（Node 内置 `node --test`），降低每次迭代的回归成本
- 种子测试阶段新增用户反馈驱动的 case（从真实用户行为中发现新的失败模式）
