## 1. 评测目标与范围

本报告回答三个问题:

1. **能不能用**: 核心路径是否闭环，用户指令是否能被正确完成。
2. **稳不稳定**: 多轮对话/边界场景下，意图与槽位是否保持一致。
3. **能不能迭代**: 是否形成可复用的评测资产(回归集 + Bad Case 库)，避免“改一次坏一次”。

评测范围覆盖:

- **调度层(Agent Router)**: 意图识别、槽位提取、工作流选择与参数传递。
- **核心端到端链路(MVP)**: 即时查询、订阅管理、推送触发等关键用户路径。

不覆盖/弱覆盖(原因是本项目以作品集交付为主，未做工程化压测):

- 压测与并发容量
- 线上监控告警体系
- 真实用户规模与增长指标

## 3. 方法论: 如何设计 Case

本项目把 Case 设计成“覆盖用户路径 + 覆盖失败模式”两条轴:

1. **用户路径覆盖**
   - 即时查询: 主题缺失追问、信源过滤、数量限制、跨语言等
   - 配置订阅: 一次性给全信息、多轮补槽位(关键词/邮箱)、模糊时间
   - 推送执行: 已配置发送、未配置引导、复合指令(先改邮箱再发送)
   - 兜底风控: Prompt 注入、无关话题

2. **失败模式覆盖**
   - 意图混淆(关键词输入被当成配置)
   - 槽位丢失/覆盖(多轮中忘记上轮信息)
   - 参数污染(关键词带冗余后缀导致检索异常)
   - 安全边界缺失(泄露系统提示词/隐私字段)
   - 性能风险(Token 过大导致延迟)

## 4. 结果与关键结论

### 4.1 调度层模型对比(18 Case)

为了避免“只看加权总分”，这里先给出更可解释的维度均分(0-10 分制)。

| 模型 | 意图精准度 | 槽位/指令遵循 | 内容质量(部分 Case) | 响应速度 |
| --- | ---:| ---:| ---:| ---:|
| DeepSeek-V3 | 10.00 | 9.17 | 9.30 | 8.56 |
| Doubao-Seed-1.6 | 10.00 | 10.00 | 8.75 | 8.61 |

结论:

- 两个模型在“意图识别”都很强，但 **调度层真正的分水岭是“槽位/指令遵循稳定性”**。
- 在本测试集中，豆包在 18/18 Case 保持槽位满分；DeepSeek 在少数 Case 出现明显丢分(从 10 掉到 5)，对调度层来说是风险项。

因此最终采用:

- 调度/扩写: 豆包 1.6
- 深度摘要/生成: DeepSeek V3.2

详见 `docs/02-模型选型分析报告.md`。

### 4.2 端到端体验的主要瓶颈与优化方向

从 Bad Case 与 PRD 复盘中，端到端体验的主要瓶颈集中在两点:

1. **延迟**: 把过多 RSS 原始内容直接交给大模型会导致超时。
2. **一致性**: 多轮补槽位时，上下文关联容易丢失。

已采用/验证有效的优化(证据见 `docs/01-PRD-产品需求文档-v2.0.md` 与 `docs/temp_sheet_4.json`):

- Top-K 预筛选(先压缩候选，再给生成模型摘要)
- 同义词/Query 改写减少输入噪音
- System Prompt 加强 `user_id` 提取与隐私边界
- 增加 Few-shot 示例与规则约束提升槽位稳定性

## 5. 代表性 Bad Case(8 例)

以下案例均来自 `docs/temp_sheet_4.json`，表述做了脱敏与概括。

| Bug ID | 关联场景 | 问题现象 | 根本原因 | 修复策略 |
| --- | --- | --- | --- | --- |
| Bug-01 | 直接看新闻 | 只回复“请等待”，未输出内容 | Prompt 路由规则缺失 | 增加专用工作流/规则，让“直接查阅”能正确进入查询链路 |
| Bug-02 | 缺少关键词 | 未追问关键词，直接进入等待话术 | 未做槽位缺失检查 | Prompt 增加“缺槽位必须追问”的硬规则 |
| Bug-03 | 内容时效性 | 发布时间输出错误 | 时区未转换、信源返回抓取时间 | 切换 RSS；代码节点统一时区；Prompt 增加时间规则 |
| Bug-05 | 响应速度 | 响应 > 40s | 输入 token 过大 | 代码节点 Top-K 预筛选；同义词库；必要时流式输出 |
| Bug-06 | 多轮补关键词 | 未成功保存上轮关键词 | 关键词提取/上下文关联弱 | 增加关键词提取规则 + 示例，提升多轮记忆一致性 |
| Bug-09 | 未配置直接发送 | 未配置却直接触发发送 | “立即发送”意图优先级过高 | 增加前置校验：未配置时先走配置引导 |
| Bug-12 | Prompt 注入 | 用户索要系统提示词时被泄露 | 缺少防御边界 | System Prompt 头部加最高优先级拒绝规则 + 话题引导 |
| Bug v1.1 | 纯关键词意图 | 用户只输入“AI”被识别为配置 | 意图边界模糊 | 增加“纯关键词=直接查询”的规则与示例 |

这些案例的共同点:

- 不是“模型不够聪明”，而是“约束不够明确/流程缺少校验”。
- 修复往往不是加长 Prompt，而是把规则、校验与工程预处理放到合适的层。

## 6. 回归策略(可持续迭代)

当前回归策略以“数据集 + 人工回归”为主，确保迭代过程可控:

1. **路由回归(18 Case)**
   - 触发条件: 调整 System Prompt、切换调度模型、调整意图表/槽位规则。
   - 验收: 槽位/指令遵循不退化；关键边界 Case(如纯关键词、未配置发送、注入攻击)必须通过。

2. **MVP 回归(22 Case)**
   - 触发条件: 工作流修改、前后端接口变更、数据结构变更。
   - 验收: 任务完成率保持 100%。

3. **Bug Bank 回归(temp_sheet_4)**
   - 原则: 每新增一个 Bad Case，都要进入长期用例库。

后续(可选)工程化方向:

- 把 18/22 Case 固化为可复现的脚本(例如 Node 内置 `node --test`)，形成最小自动化回归。
