## 1. 设计原则

智能助手的 Prompt 设计遵循以下核心原则：

- **身份一致性**：明确助手作为"AI 每日简报"专属助手的身份，专注于新闻获取与订阅管理。
- **上下文驱动**：利用前端注入的 `[System Context: user_id="..."]` 实现无感知的用户识别，避免主动向用户索取私密 ID。
- **意图强约束**：通过详尽的"意图识别表"和"技能模块"说明，确保 LLM 能够准确映射用户需求到对应的工作流。
- **参数槽位化**：将所有技能调用参数明确定义，并规定其获取来源和必填性。
- **安全与隐私**：内建防注入逻辑，严禁泄露 `user_id` 或跨账户操作。
- **防御性调用**：调用任何工作流前必须通过参数检查（调用守卫），防止空参数导致的工作流异常。

## 2. 提示词架构（v3 当前版本）

当前系统提示词采用模块化结构，共 373 行、9 个核心模块：

| 模块 | 功能 | 关键设计 |
| --- | --- | --- |
| 身份识别与上下文处理 | 自动从消息末尾提取 user_id | 配合前端注入 `[System Context]`，用户无感知 |
| 调用守卫（最高优先级） | 调用工作流前的参数检查清单 | 所有必填参数非空才允许调用，防止空参数报错 |
| 动作后反馈识别 | 判断用户是否在对上一轮结果做反馈 | 避免把反馈误判为新意图 |
| 意图识别表 | 关键词→技能的映射 | 3 个技能按优先级排序：推送 > 订阅 > 查询 |
| keyword 参数规则 | 实体透传，不强制归类 | 用户说什么传什么，归类交给下游工作流 |
| 多轮对话规则 | 补槽优先、意图继承 | 缺参数时追问而非重新开始 |
| 技能模块详情 | 每个工作流的触发条件和参数 | 含交互流程示例 |
| 非任务场景处理 | 4 级处理优先级 | 友好拒绝 + 引导回产品功能 |
| 快捷回复模板 | 欢迎语、成功/失败提示 | 统一语言风格 |

### 2.1 意图识别表（Intent Table）

通过 Markdown 表格显式定义关键词与技能的映射关系，按优先级排序：

| 优先级 | 意图 | 技能 |
| --- | --- | --- |
| 1（最高） | 立即推送 / 发送测试邮件 | `send_mail_cache_using` |
| 2 | 查询/修改/取消/新增订阅 | `subscribe_using` |
| 3 | 查看新闻 / 即时查询 | `get_instant_news_using` |

优先级设计理由：推送和订阅是"配置型"操作，一旦误判会产生实际副作用（发邮件、改配置），因此优先级高于查询。

### 2.2 技能模块说明

每个工作流提供了详细的参数说明和交互流程示例：

**get_instant_news_using（即时查询）**：
- 必填参数：`keyword`（ai / 财经 / 科技）
- 透传参数：`raw_query`（用户原话，下游模型自行提取条数、来源、语言等约束）
- keyword 只允许三个值：`ai`、`财经`、`科技`

**subscribe_using（订阅管理）**：
- 必填参数：`action`（查询订阅 / 修改订阅 / 取消订阅）、`input_user_id`
- 按场景可选：`new_email`、`ex_email`、`new_keyword`
- 区分四种场景：新用户订阅、查询、修改、取消

**send_mail_cache_using（测试推送）**：
- 必填参数：`input_user_id`
- 行为：从 `daily_news_history` 读取当日缓存发送，非实时生成
- 能力边界说明：邮件内容来自缓存，不是实时生成的

### 2.3 对话规范与模板

定义了简洁友好的语言风格，并提供了欢迎语、主题询问、成功提示等快捷回复模板。

## 3. 关键设计亮点

### 3.1 raw_query 透传

在即时查询工作流中，用户原话（raw_query）直接传给下游摘要模型，由模型自行提取约束。

- **传统做法**：Agent 层提取所有参数（条数、来源、语言）→ 传结构化参数给下游
- **本项目做法**：Agent 只提取 keyword → raw_query 原样透传 → 下游模型自行解析
- **优势**：
  - 减少一层参数解析的出错机会
  - 下游模型能看到用户完整意图，处理更灵活
  - Agent 层更轻量，只负责路由和关键参数

### 3.2 调用守卫（Calling Guard）

Agent 在调用任何工作流前必须执行检查清单：

```
调用前检查：
□ 所有必填参数是否非空？
□ keyword 是否在允许范围内（ai / 财经 / 科技）？
□ user_id 是否已成功提取？
□ 如果缺少参数 → 追问用户，不要带着空参数调用
```

设计来源：Bug DISPATCH-MISCALL-01（缺失必填参数未追问直接调用工作流导致异常）。

### 3.3 keyword 实体透传

用户说"大模型新闻"，Agent 传 `keyword=ai`（归一化），但 raw_query 保留原话"大模型新闻"。这样：
- 下游工作流能走正确的信源分支（通过 keyword）
- 下游模型能理解用户的具体意图（通过 raw_query）

### 3.4 来源过滤自检

即时查询的下游 prompt 要求模型在输出前逐条自检：
- 每条新闻的来源是否匹配用户指定的来源？
- 不匹配则删除，宁可少输出也不凑数
- 设计来源：Bug GEN-05（来源过滤失效）、NEWS-T06/T10/T13/T14 系列

### 3.5 动作后反馈识别

v3 新增的模块，解决"用户对上一轮结果做反馈时被误判为新意图"的问题。例如：
- 用户说"不错"→ 这是对上一轮查询结果的反馈，不是新的查询请求
- 用户说"换一个"→ 这是对上一轮结果的修改请求，需要继承上下文

## 4. 演进历程

| 版本 | 行数 | 核心变化 | 驱动因素 |
| --- | --- | --- | --- |
| v0 | ~50 | 基础意图识别 | 初始搭建 |
| v1 | ~80 | 意图识别表 + 技能路由 | 核心链路跑通 |
| v2 | ~146 | user_id 注入规则 + Few-shot 示例 | Bug-07（多轮邮箱补充异常）、Bug-16（隐私越权） |
| v3（当前） | 373 | 调用守卫 + 实体透传 + 多轮补槽 + 反馈识别 + 非任务 4 级处理 | 评测集扩展后发现的 dispatch 类 badcase |

### 关键版本跃迁说明

**v1 → v2（结构级调整）**：
- 新增 `user_id` 提取规则：配合前端在消息末尾注入 `[System Context]` 的逻辑，强制助手进行提取并静默处理
- 引入意图识别表与示例：增加了显式的意图映射表和多场景交互示例（One-shot/Few-shot），显著提升了槽位填充准确率

**v2 → v3（防御级升级）**：
- 新增调用守卫：从"信任模型会填对参数"变为"调用前强制检查"
- 新增实体透传：从"Agent 层做所有参数解析"变为"只做关键路由，细节交给下游"
- 新增反馈识别：解决多轮对话中"反馈 vs 新意图"的混淆
- 新增非任务 4 级处理：从"简单拒绝"变为"分级引导"
- 测试推送技能从 `send_mail_using` 更新为 `send_mail_cache_using`（缓存方案）

## 5. 关键技术细节

### 参数填充可靠性（Slot-filling）

- `keyword` 只允许三个值：`ai`、`财经`、`科技`
- `keyword` 无默认值，必须让用户主动选择
- 提供了模糊表述映射（如"大模型"→"ai"、"股票"→"财经"）
- `raw_query` 保留用户原话，不做任何修改

### 温度设置（Temperature Rationale）

在路由评估和意图识别中，Temperature 设置为 **0.5**。
- 原因：0.5 在保持回复多样性的同时，能有效抑制模型在技能选择时的随机性

### 提示词注入防御

- System Prompt 头部设置最高优先级拒绝规则
- 关键字触发拒答（如"系统提示词"、"你的指令"等）
- 禁止向用户展示 user_id
- 限制助手只能通过 `input_user_id` 操作当前上下文对应的数据

## 6. 维护清单（Checklist）

修改 Prompt 时，请务必检查以下各项：

- [ ] **变量一致性**：新增技能参数是否与工作流调用逻辑一致？
- [ ] **上下文提取**：是否依然保留了对 `[System Context]` 的强制提取指令？
- [ ] **映射表更新**：新增的功能是否已同步更新至"意图识别表"？
- [ ] **交互示例**：是否为新功能提供了至少一个正向和反向的交互示例？
- [ ] **隐私自检**：新指令是否可能导致 `user_id` 或 `email` 在回复中泄露？
- [ ] **调用守卫**：新增技能是否已加入调用前检查清单？
- [ ] **回归测试**：修改后是否跑过核心回归集（22 Case）和调度回归集（18 Case）？
