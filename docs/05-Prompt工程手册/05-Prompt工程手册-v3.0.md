## 1. 设计原则

智能助手的 Prompt 设计遵循以下核心原则：

- **身份一致性**：明确助手作为"AI 每日简报"专属助手的身份，专注于新闻获取与订阅管理。
- **上下文驱动**：利用前端注入的 `[System Context: user_id="..."]` 实现无感知的用户识别，避免主动向用户索取私密 ID。
- **意图强约束**：通过详尽的"意图识别表"和"技能模块"说明，确保 LLM 能够准确映射用户需求到对应的工作流。
- **参数槽位化**：将所有技能调用参数明确定义，并规定其获取来源和必填性。
- **安全与隐私**：内建防注入逻辑，严禁泄露 `user_id` 或跨账户操作。
- **防御性调用**：调用任何工作流前必须通过参数检查（调用守卫），防止空参数导致的工作流异常。

## 2. 提示词架构（v3 当前版本）

当前系统提示词采用模块化结构，共 373 行、9 个核心模块：

| 模块 | 功能 | 关键设计 |
| --- | --- | --- |
| 身份识别与上下文处理 | 自动从消息末尾提取 user_id | 配合前端注入 `[System Context]`，用户无感知 |
| 调用守卫（最高优先级） | 调用工作流前的参数检查清单 | 所有必填参数非空才允许调用，防止空参数报错 |
| 动作后反馈识别 | 判断用户是否在对上一轮结果做反馈 | 避免把反馈误判为新意图 |
| 意图识别表 | 关键词→技能的映射 | 3 个技能按优先级排序：推送 > 订阅 > 查询 |
| keyword 参数规则 | 实体透传，不强制归类 | 用户说什么传什么，归类交给下游工作流 |
| 多轮对话规则 | 补槽优先、意图继承 | 缺参数时追问而非重新开始 |
| 技能模块详情 | 每个工作流的触发条件和参数 | 含交互流程示例 |
| 非任务场景处理 | 4 级处理优先级 | 友好拒绝 + 引导回产品功能 |
| 快捷回复模板 | 欢迎语、成功/失败提示 | 统一语言风格 |

### 2.1 意图识别表（Intent Table）

通过 Markdown 表格显式定义关键词与技能的映射关系，按优先级排序：

| 优先级 | 意图 | 技能 |
| --- | --- | --- |
| 1（最高） | 立即推送 / 发送测试邮件 | `send_mail_cache_using` |
| 2 | 查询/修改/取消/新增订阅 | `subscribe_using` |
| 3 | 查看新闻 / 即时查询 | `get_instant_news_using` |

优先级设计理由：推送和订阅是"配置型"操作，一旦误判会产生实际副作用（发邮件、改配置），因此优先级高于查询。

### 2.2 技能模块说明

每个工作流提供了详细的参数说明和交互流程示例：

**get_instant_news_using（即时查询）**：
- 必填参数：`keyword`（ai / 财经 / 科技）
- 透传参数：`raw_query`（用户原话，下游模型自行提取条数、来源、语言等约束）
- keyword 只允许三个值：`ai`、`财经`、`科技`

**subscribe_using（订阅管理）**：
- 必填参数：`action`（查询订阅 / 修改订阅 / 取消订阅）、`input_user_id`
- 按场景可选：`new_email`、`ex_email`、`new_keyword`
- 区分四种场景：新用户订阅、查询、修改、取消

**send_mail_cache_using（测试推送）**：
- 必填参数：`input_user_id`
- 行为：从 `daily_news_history` 读取当日缓存发送，非实时生成
- 能力边界说明：邮件内容来自缓存，不是实时生成的

### 2.3 对话规范与模板

定义了简洁友好的语言风格，并提供了欢迎语、主题询问、成功提示等快捷回复模板。

## 3. 关键设计亮点

### 3.1 raw_query 透传

在即时查询工作流中，用户原话（raw_query）直接传给下游摘要模型，由模型自行提取约束。

- **传统做法**：Agent 层提取所有参数（条数、来源、语言）→ 传结构化参数给下游
- **本项目做法**：Agent 只提取 keyword → raw_query 原样透传 → 下游模型自行解析
- **优势**：
  - 减少一层参数解析的出错机会
  - 下游模型能看到用户完整意图，处理更灵活
  - Agent 层更轻量，只负责路由和关键参数

### 3.2 调用守卫（Calling Guard）

Agent 在调用任何工作流前必须执行检查清单：

```
调用前检查：
□ 所有必填参数是否非空？
□ keyword 是否在允许范围内（ai / 财经 / 科技）？
□ user_id 是否已成功提取？
□ 如果缺少参数 → 追问用户，不要带着空参数调用
```

设计来源：Bug DISPATCH-MISCALL-01（缺失必填参数未追问直接调用工作流导致异常）。

### 3.3 keyword 实体透传

用户说"查看人工智能新闻"，Agent 通过 `KEYWORD_ALIASES` 将"人工智能"归一化为 `keyword=ai`，同时 `raw_query` 保留用户原话"查看人工智能新闻"。这样：
- 下游工作流能走正确的信源分支（通过 keyword）
- 下游模型能理解用户的具体意图（通过 raw_query）

### 3.4 来源过滤自检

即时查询的下游 prompt 要求模型在输出前逐条自检：
- 每条新闻的来源是否匹配用户指定的来源？
- 不匹配则删除，宁可少输出也不凑数
- 设计来源：Bug GEN-05（来源过滤失效）、NEWS-T06/T10/T13/T14 系列

### 3.5 动作后反馈识别

v3 新增的模块，解决"用户对上一轮结果做反馈时被误判为新意图"的问题。例如：
- 用户说"不错"→ 这是对上一轮查询结果的反馈，不是新的查询请求
- 用户说"换一个"→ 这是对上一轮结果的修改请求，需要继承上下文

## 4. 演进历程

| 版本 | 行数 | 核心变化 | 驱动因素 |
| --- | --- | --- | --- |
| v0 | ~50 | 基础意图识别 | 初始搭建 |
| v1 | ~80 | 意图识别表 + 技能路由 | 核心链路跑通 |
| v2 | ~146 | user_id 注入规则 + Few-shot 示例 | Bug-07（多轮邮箱补充异常）、Bug-16（隐私越权） |
| v3（当前） | 373 | 调用守卫 + 实体透传 + 多轮补槽 + 反馈识别 + 非任务 4 级处理 | 评测集扩展后发现的 dispatch 类 badcase |

### 关键版本跃迁说明

**v1 → v2（结构级调整）**：
- 新增 `user_id` 提取规则：配合前端在消息末尾注入 `[System Context]` 的逻辑，强制助手进行提取并静默处理
- 引入意图识别表与示例：增加了显式的意图映射表和多场景交互示例（One-shot/Few-shot），显著提升了槽位填充准确率

**v2 → v3（防御级升级）**：
- 新增调用守卫：从"信任模型会填对参数"变为"调用前强制检查"
- 新增实体透传：从"Agent 层做所有参数解析"变为"只做关键路由，细节交给下游"
- 新增反馈识别：解决多轮对话中"反馈 vs 新意图"的混淆
- 新增非任务 4 级处理：从"简单拒绝"变为"分级引导"
- 测试推送技能从 `send_mail_using` 更新为 `send_mail_cache_using`（缓存方案）

## 5. 关键技术细节

### 参数填充可靠性（Slot-filling）

- `keyword` 只允许三个值：`ai`、`财经`、`科技`
- `keyword` 无默认值，必须让用户主动选择
- `raw_query` 保留用户原话，不做任何修改

### 温度设置（Temperature Rationale）

在路由评估和意图识别中，Temperature 设置为 **0.5**。
- 原因：0.5 在保持回复多样性的同时，能有效抑制模型在技能选择时的随机性

### 提示词注入防御

- System Prompt 头部设置最高优先级拒绝规则
- 关键字触发拒答（如"系统提示词"、"你的指令"等）
- 禁止向用户展示 user_id
- 限制助手只能通过 `input_user_id` 操作当前上下文对应的数据

## 6. 即时查询提示词设计

即时查询工作流包含两个模型节点：关键词扩写（豆包 1.6 极致速度）和生成提示词（DeepSeek V3.2）。

### 6.1 关键词扩写（豆包 1.6 极致速度）

将用户输入的 keyword 扩展为能最大化命中新闻标题的实体词组，供下游 RSS 检索使用。

**核心目标**：最大化召回率——宁可多捞，不可漏掉。

**输入/输出**：
- 输入：`{{keyword}}`（单词或逗号分隔的多词）
- 输出：纯 JSON 数组 `["词1", "词2", ...]`，禁止 Markdown 标记和解释文字

**扩写规则**：

| 词类型 | 判定条件 | 扩写数量 |
| --- | --- | --- |
| 热门短尾词 | ≤2 字或 ≤4 字母 | 12-15 个 |
| 普通词 | 3-4 字常见名词 | 8-10 个 |
| 长尾词 | ≥5 字，或含年份 | 6-8 个 |

**扩写方向（按优先级）**：
1. 中英文互补：同一概念的中英文都覆盖（AI ↔ 人工智能）
2. 头部品牌/人物：新闻常提及的主体名称
3. 核心产品/技术：该领域的热门细分
4. 泛化品类词：兜底用的上位词

**关键约束**：
- 禁止照抄示例，必须动态生成
- 反映当前热点，优先扩展当前活跃的品牌和产品
- 品牌优先于品类（新闻标题更常出现"华为"而非"手机厂商"）
- 长尾词先去掉修饰语（年份、"新款"等），提取核心实体后再扩写

**设计思路**：
- 即时查询版比日报版更精细，按词类型分配数量、按优先级排列扩写方向
- 原因：即时查询面对的用户输入更多样（可能是品牌名、技术术语、长尾描述），需要更细粒度的召回策略

### 6.2 生成提示词（DeepSeek V3.2）

对 RSS 抓取回来的新闻列表进行筛选、聚类、排序和格式化输出。这是即时查询的核心生成节点。

**输入参数**：
- `{{raw_query}}`：用户原话（最高优先级约束来源）
- `{{keyword}}`：归一化后的主题词
- `{{current_time}}`：当前时间

**6 步执行流程**：

| 步骤 | 功能 | 关键设计 |
| --- | --- | --- |
| Step 1 | 理解用户意图 | 从 raw_query 识别来源约束、条数限制、语言偏好；含来源别名映射表（36kr→36氪 等） |
| Step 2 | 来源过滤 | 白名单机制（8 个信源），用户指定来源时严格过滤，不在白名单则输出空状态 |
| Step 3 | 时效过滤 | 仅保留 current_time 往前 24 小时内的新闻 |
| Step 4 | 语义聚类与排序 | 同一事件归组取代表；按报道数→跨媒体类型→时间三级排序 |
| Step 4.5 | 输出前自检 | 排序逐对检查、🔥一致性检查、热度行格式检查 |
| Step 5 | 数量控制 | 用户指定则遵守（上限 10 条），未指定默认 5 条 |
| Step 6 | 语言输出 | raw_query 含英文关键词则英文输出，否则中文 |

**核心设计亮点**：

**① 来源过滤自检（双重保障）**
- 用户指定来源时，输出前逐条检查每条新闻的来源字段是否精确匹配
- 不匹配则立即删除，即使删除后条数不足也绝不用其他来源补充
- 设计来源：Bug GEN-05（来源过滤失效）

**② 热度标记体系**
- 基于聚类结果中的报道数量和跨媒体类型数量，同时决定 🔥 标记和热度行文案
- ≥3 家 + ≥2 类 = 🔥🔥🔥，≥2 家 + ≥2 类 = 🔥🔥，≥2 家同类 = 🔥，1 家 = 无标记
- 热度行和 🔥 标记必须一致，输出前强制自检

**③ 排序自检（逐对比较）**
- 不依赖模型的"直觉排序"，要求逐对比较相邻条目
- 如果发生交换则从头再检查一遍（冒泡排序思路）
- 确保排序结果严格符合统计数据而非主观判断

**④ 来源别名映射**
- 用户可能用非标准名称指定来源（如 36kr、ithome、wallstreetcn）
- 提示词内置映射表，将非标准名转为系统标准名后再过滤

**⑤ 防误解规则**
- "100 条 AI 新闻"中的"100 条"是条数请求，不是来源名
- 提示词显式声明数字+"条"的语义，防止模型误判

**反向示例（❌ 禁止行为）**：
- 不输出非指定来源的新闻
- 不超出条数限制（上限 10 条）
- 不用无关新闻凑数
- 不添加额外说明文字
- 不混用中英文
- 不省略格式字段

## 7. 推送生成提示词设计

定时推送工作流包含两个模型节点：关键词扩写（豆包 1.6）和主题生成提示词（DeepSeek V3.2）。三个主题（AI / 科技 / 财经）各有独立的生成提示词。

### 7.1 关键词扩写（豆包 1.6）

与即时查询版的关键词扩写功能相同，但设计更简洁。

**与即时查询版的差异**：

| 维度 | 即时查询版 | 日报版 |
| --- | --- | --- |
| 数量分配 | 按词类型分级（12-15 / 8-10 / 6-8） | 统一 15-20 个 |
| 扩写方向 | 4 级优先级排列 | 同义词 + 下位词 + 关联词（无优先级） |
| 长尾词处理 | 显式去修饰语规则 | 无特殊处理 |
| 约束 | 禁止照抄 + 反映热点 + 品牌优先 | 中英文必须同时出现（ai/人工智能） |

**设计思路**：日报版的输入是固定的三个主题词（ai / 科技 / 财经），变化较少，不需要即时查询版那样精细的分类策略。

### 7.2 主题生成提示词（DeepSeek V3.2）

三个主题共享相同的提示词框架，但在相关性过滤、排序规则、信源分级上做了差异化。

**共享框架**：

| 模块 | 功能 | 三主题通用 |
| --- | --- | --- |
| 语义去重 | 同一事件判定 + 保留版本优先级 + 去重自检 | ✅ |
| 相关性过滤 | 删除与 keyword 不直接相关的新闻 | 规则差异化 |
| 排序规则 | 按新闻价值星级排序 | 星级定义差异化 |
| 信源选择 | 三原则：独家优先→专业→内容丰富度 | 信源分级差异化 |
| 信源多样性 | 单一信源占比上限 | 阈值差异化 |
| 摘要规则 | 极简主义，硬上限 120 字 | ✅ |
| 数量控制 | top_news + quick_news 分级输出 | ✅ |
| 输出前自检 | 去重 + 排序 + 遗漏 + 信源占比 + 摘要字数 + 格式 | ✅ |

**三主题差异化设计**：

#### 相关性过滤差异

| 主题 | 过滤规则 | 设计意图 |
| --- | --- | --- |
| AI | 删除与 AI 不直接相关的新闻 | 纯 AI 聚焦 |
| 科技 | 纯 AI 新闻占比不超过 30%；优先保留芯片、手机、消费电子等硬科技 | 与 AI 主题差异化，避免内容重叠 |
| 财经 | 严格边界定义：读者看完反应是"调整持仓"→收录，"技术好酷"→不收录 | 聚焦金融市场视角 |

科技主题的 AI 内容比例控制是一个关键设计：因为已有独立的 AI 主题，科技主题需要主动控制纯 AI 内容占比，但 AI + 硬件的交叉新闻（如"AI 芯片量产"）算科技不算纯 AI。

#### 排序星级差异

| 主题 | ⭐⭐⭐⭐⭐ 示例 | 特有规则 |
| --- | --- | --- |
| AI | 行业格局变化、重磅产品发布、独家爆料 | — |
| 科技 | 行业格局变化、重磅产品发布、独家爆料 | 与 AI 主题相似但侧重硬件/消费电子 |
| 财经 | 重大政策/监管、市场重大波动、头部企业重大事件 | 含排序自检：⭐⭐⭐⭐⭐ 是否排在 ⭐⭐⭐ 之前；当日重大行情是否遗漏 |

#### 信源分级差异

| 主题 | 一级信源 | 二级信源 | 三级信源 |
| --- | --- | --- | --- |
| AI | 36氪、IT之家、量子位、AIbase、机器之心 | 虎嗅、钛媒体、快科技、爱范儿、少数派 | 华尔街见闻、界面 |
| 科技 | 36氪、极客公园、IT之家、快科技、The Verge、TechCrunch | 澎湃科技、新浪科技、腾讯科技等 | 华尔街见闻、财联社等 |
| 财经 | 华尔街见闻、财联社、第一财经、财新网 | 格隆汇、界面 | 钛媒体 |

信源分级直接影响"保留版本优先级"：同一事件被多家报道时，一级信源的版本优先保留。

#### 信源多样性阈值差异

| 主题 | 单一信源占比上限 |
| --- | --- |
| AI | 30% |
| 科技 | 30% |
| 财经 | 50% |

财经主题阈值更宽松，因为财经核心信源较少（华尔街见闻、财联社等），过严会导致有价值的新闻被误删。

### 7.3 核心设计亮点

**① 语义去重三步自检**
- 不仅在筛选阶段去重，还要求模型在输出前对 top_news 和 quick_news 做交叉检查
- top_news 内部逐对比较 → quick_news 内部逐对比较 → top_news × quick_news 交叉比较
- 若重复，删除信源匹配度低的那条（top_news × quick_news 重复时删 quick_news 中的）

**② 同一事件判定规则（AI 主题特有）**
- AI 主题的提示词显式定义了"同一事件"的判定条件：
  - 标题/摘要中出现相同的产品名/型号
  - 同一公司 + 同类动作
  - 引用了相同的核心数字
- 并特别提醒：不同媒体标题写法差异很大，要看底层事实是否相同

**③ 摘要字数的"故意压低"策略**
- 目标设为 100-120 字，但提示词注释说明"你通常会写超，所以目标故意压低"
- 这是一种针对 LLM 输出倾向的补偿策略：已知模型倾向于写长，因此将目标值设低于实际期望
- 配合"写完后逐字计数，超过 120 字必须删减"的硬约束

**④ 输出格式为纯 JSON**
- 推送生成的输出是结构化 JSON（top_news + quick_news），不是最终用户看到的内容
- 因此提示词强调"纯 JSON，禁止 Markdown 代码块"，防止代码节点解析失败

**⑤ 反向示例的 Badcase 驱动**
- 三个主题的提示词中都包含大量反向示例（❌ 禁止行为）
- 这些反向示例均来自实际 badcase 的修复，不是凭空设计的
- 例如：信源多样性规则来自某次输出中单一信源占比过高的问题

## 8. 维护清单（Checklist）

修改 Prompt 时，请务必检查以下各项：

- [ ] **变量一致性**：新增技能参数是否与工作流调用逻辑一致？
- [ ] **上下文提取**：是否依然保留了对 `[System Context]` 的强制提取指令？
- [ ] **映射表更新**：新增的功能是否已同步更新至"意图识别表"？
- [ ] **交互示例**：是否为新功能提供了至少一个正向和反向的交互示例？
- [ ] **隐私自检**：新指令是否可能导致 `user_id` 或 `email` 在回复中泄露？
- [ ] **调用守卫**：新增技能是否已加入调用前检查清单？
- [ ] **回归测试**：修改后是否跑过核心回归集（22 Case）和调度回归集（18 Case）？
