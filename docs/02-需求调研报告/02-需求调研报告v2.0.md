## 0. 调研背景

### 为什么做这个产品

作为一个对 AI 行业高度关注的从业者，我每天需要花大量时间在多个信源之间切换来获取行业资讯：刷 36氪快讯、翻 Twitter/X 上的 AI 大V、看 Hacker News、逛即刻的 AI 圈子……信息分散在十几个平台，重复内容多、噪音大，每天光是"找新闻"就要花 30-60 分钟，但真正有价值的信息可能只需要 5 分钟就能读完。

这个痛点不是我一个人的。在和朋友、同事的日常交流中，经常听到类似的抱怨："每天信息太多了，根本看不过来"、"总怕漏掉什么重要的事"。

与此同时，AI 技术（特别是大语言模型）的成熟让"自动化信息聚合+摘要"成为可能。我开始思考：能不能做一个工具，帮我（以及和我有同样痛点的人）每天用 5 分钟搞定行业资讯？

### 调研目标

1. 验证"信息过载"痛点是否普遍存在，而非仅是个人感受
2. 了解目标用户当前的信息获取习惯和不满
3. 分析现有解决方案的优劣，寻找差异化空间
4. 形成产品假设，为 MVP 开发提供方向

## 1. 调研方法

本次调研采用"精益调研"方法——作为个人 side project，没有专业调研团队和预算，但通过多种轻量方式收集了有效信息。

| 方法                   | 具体做法                                                                                                                                                  | 产出                         |
| ---------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------- | ---------------------------- |
| **自身体验分析**       | 记录自己一周内的信息获取行为：用了哪些平台、花了多少时间、哪些信息真正有价值                                                                              | 痛点清单、行为模式           |
| **非正式用户沟通**     | 与身边对 AI/科技/财经感兴趣的朋友、同事交流信息获取习惯和痛点                                                                                             | 需求验证、场景补充           |
| **社区调研（Reddit）** | 通过 Perplexity 搜索 Reddit（r/productivity、r/artificial、r/ChatGPT、r/devops 等）上关于"AI newsletter"、"news aggregator"、"information overload"的讨论 | 英文用户真实声音、痛点结构化 |
| **社区调研（知乎）**   | 搜索知乎近三年关于信息过载、AI 摘要工具、知识管理的讨论                                                                                                   | 中文用户视角、信任顾虑       |
| **竞品体验**           | 订阅并使用 TLDR、The Rundown AI、Feedly、Perplexity Discover 等产品 1-2 周                                                                                | 功能对比、体验差距           |
| **市场数据收集**       | 搜索 Newsletter 赛道的增长数据、AI 新闻产品的融资和用户量                                                                                                 | 市场验证                     |

## 2. 核心发现

### 2.1 痛点验证：信息过载是普遍问题

**自身体验**：

- 每天在 5-8 个平台之间切换获取 AI/科技资讯
- 实际阅读时间 30-60 分钟，但有价值的信息只需 5-10 分钟
- 经常出现"刷了半小时，感觉什么都没记住"的情况
- FOMO（错失恐惧）：不刷就担心漏掉重要事件

**Reddit 社区调研发现**：

Reddit 上关于信息过载的讨论非常活跃，用户痛点可归纳为四个核心主题：

**主题一：信息噪音与重复轰炸**

用户不仅被信息淹没，更被低质量的重复信息折磨。

> "现在的 AI 新闻源太糟糕了，我经常在不同的地方把同一个故事读上 10 遍，却以为是新东西。"

> "为了跟上 DevOps 行业的动态，我不得不每天手动检查几十个网站，这感觉完全是浪费生命。"

> "Google 的搜索结果和新闻流里充满了垃圾 SEO 内容和广告，根本找不到真正有价值的讨论。"

**主题二：FOMO（错失恐惧症）**

用户不停刷新新闻流并非因为喜欢阅读，而是因为恐惧——担心被快速变化的行业抛下。

> "我有严重的 FOMO，总觉得如果不每小时刷一次 Reddit 或 Twitter，就会错过下一个 ChatGPT 级别的突破。"

> "我不需要更多的内容，我需要一个工具告诉我'今天你只需要知道这三件事'，然后让我安心去工作。"

> "行业变化太快了，我感觉自己像是在做第二份全职工作，仅仅是为了不掉队。"

**主题三：信任危机与幻觉焦虑**

这是阻碍专业用户全面转向 AI 摘要工具的最大心理障碍。

> "我试过用 AI 总结技术文章，但它经常编造不存在的代码库或事实（Hallucinations），这对我的工作来说是致命的。"

> "我想要一个能打破回音室的工具，而不是算法觉得我喜欢什么就只给我推什么。"

**主题四：工具碎片化与体验断层**

现有工具要么太老旧（原始 RSS），要么太复杂（开发者工具），要么太封闭（算法推荐），缺乏"中间地带"。

> "RSS 阅读器（如 Feedly）虽然好，但还是让我面对成百上千个未读标题，压力太大了。"

> "我不得不自己写 Python 脚本和 LangChain 来抓取新闻，因为市面上没有现成的工具能满足我的定制需求。"

**知乎社区调研发现**：

知乎上的讨论更侧重于中文用户的视角，痛点有所不同：

| 主题                     | 核心发现                                                                 |
| ------------------------ | ------------------------------------------------------------------------ |
| 时间与效率焦虑           | 用户普遍感到无效的信息筛选挤压了深度思考的时间，导致焦虑和产出质量下降   |
| 信息过载与筛选成本       | 个人知识库膨胀后难以管理和检索，用户需要的是"信息过滤器"而非更多信息     |
| 对技术信任与可控性的担忧 | 即使利用 AI 工具，用户强调人的主导性和对过程的控制，担心"黑盒"工具的偏见 |
| 个性化与精准性的高阶需求 | 通用化信息无法满足专业需求，用户追求超越平均水平的深度和针对性           |

**关键洞察**：用户的核心需求不是"更多信息"，而是"更少但更精准的信息"。降噪 > 增量。

### 2.2 现有方案的不足

通过竞品体验、社区口碑和 Reddit 用户反馈，总结了现有方案的主要问题：

| 现有方案                                                 | 用户常见抱怨                                                 | 未满足的需求                   |
| -------------------------------------------------------- | ------------------------------------------------------------ | ------------------------------ |
| **RSS 阅读器**（Feedly、Inoreader、NewsBlur）            | "配置太麻烦"、"信息量还是太大"、"成百上千个未读标题压力太大" | 开箱即用、自动筛选、语义去重   |
| **英文 Newsletter**（TLDR、The Rundown AI）              | "只有英文"、"不能按需查询"、"有数天的滞后"                   | 中文支持、交互能力、实时性     |
| **新闻 App**（Google News、今日头条）                    | "推荐算法不准"、"信息茧房"、"SEO 垃圾内容多"                 | 透明信源、主题可控、无广告     |
| **大模型搜索**（Perplexity、ChatGPT）                    | "每次都要主动搜"、"没有推送"、"AI 幻觉"                      | 定时推送、订阅管理、可验证来源 |
| **科技媒体**（36氪、虎嗅）                               | "内容太多，还是过载"、"广告多"                               | 精简摘要、无广告               |
| **社区与社交媒体**（Hacker News、Reddit、Twitter Lists） | "信噪比不稳定"、"需要大量时间筛选"                           | 自动聚合、结构化输出           |
| **DIY 方案**（Python 脚本、LangChain）                   | "门槛太高"、"维护成本大"                                     | 开箱即用的产品化方案           |

**关键洞察**：市场上缺少一个"中文 + 自动推送 + 可交互 + 精简摘要"的产品。

### 2.3 用户行为模式

通过自身体验和非正式沟通，总结了两种典型的信息获取模式：

**模式 A：定时浏览型**

- 固定时间（早上通勤、午休）集中浏览资讯
- 偏好"推送到手边"而非"主动去找"
- 对邮件/Newsletter 接受度高
- 核心诉求：每天 5 分钟看完当日要闻

**模式 B：即时查询型**

- 有具体需求时才查（开会前、写报告前）
- 需要快速获取某个话题的最新动态
- 偏好对话式交互（像问人一样问 AI）
- 核心诉求：问一句话，30 秒内得到结构化答案

**关键洞察**：两种模式不是互斥的，同一个用户在不同场景下会切换。产品应该同时支持"定时推送"和"即时查询"。

### 2.4 市场机会信号

**Newsletter 赛道的增长**：

- TLDR Newsletter 在 2023-2024 年从 50 万增长到 120 万+ 订阅
- The Rundown AI 在不到一年内达到 60 万+ 订阅
- Morning Brew 以约 7500 万美元被收购
- 这些数据说明"邮件+精选资讯"模式有强劲的用户需求

**中文市场空白**：

- 英文 AI Newsletter 赛道已有多个成熟产品
- 中文市场没有同等定位的产品（36氪/虎嗅是媒体，不是 AI 聚合工具）
- 国内大模型产品（Kimi、智谱）的新闻能力是附属功能，不够专业

### 2.5 用户核心任务（JTBD 分析）

基于 Reddit 和知乎的社区调研，提炼出目标用户的三条核心待办任务：

**JTBD 1：高效"晨报"与去噪**

> "当我在早晨开始工作或利用碎片时间补课时，我想快速浏览一份经过去重、去除噪音的行业核心动态摘要，从而在 5-10 分钟内掌握全貌，把剩下的时间留给真正的工作。"

- 现有方案：RSS 阅读器手动扫描标题、社交媒体列表浏览
- 主要不足：严重的信息重复、信噪比低、线性处理效率低

**JTBD 2：可信的信息验证**

> "当我需要引用或转发某条 AI 新闻时，我想获得带原文链接的摘要，从而快速验证事实准确性，不必担心被 AI 幻觉误导。"

- 现有方案：通用 LLM 直接询问、手动多方搜索对比
- 主要不足：AI 幻觉风险、验证流程繁琐、访问障碍（付费墙）

**JTBD 3：消除 FOMO 的"背景监控"**

> "当我沉浸在深度工作无法实时在线时，我想拥有一个系统在后台监控我的关注领域，并在每天固定时间推送摘要，从而彻底消除'错失恐惧症'，获得不用时刻刷新的心理安全感。"

- 现有方案：行业周刊/Newsletter、强迫性刷新社交媒体
- 主要不足：Newsletter 有数天滞后、实时推送太多太杂、心理焦虑未解

### 2.6 用户担忧与信任障碍

社区调研中发现，即使用户认可产品价值，仍存在以下深层顾虑：

| 担忧         | 具体表现                                                     | 对产品的启示                         |
| ------------ | ------------------------------------------------------------ | ------------------------------------ |
| **AI 幻觉**  | 担心 AI 编造事实或错误总结，对需要准确信息的专业人士是致命的 | 每条摘要必须带原文链接，让用户可验证 |
| **信息茧房** | 担心算法只推送符合口味的内容，视野被窄化                     | 信源透明、主题可控，不做个性化推荐   |
| **过度过滤** | 担心算法不仅过滤了噪音，也误杀了关键的"弱信号"               | 提供信源白名单控制，不过度筛选       |
| **隐私泄露** | 警惕需要跟踪阅读行为的工具                                   | 最小化数据收集，不做广告画像         |
| **能力退化** | 担心习惯了"嚼碎的流食"后丧失独立判断能力                     | 摘要是入口而非终点，鼓励点击原文     |

**关键洞察**：信任是此类产品从"玩具"走向"工具"的关键。NewsFlow.ai 的设计选择——每条新闻带原文链接、信源透明可追溯、不做个性化推荐——直接回应了这些担忧。

## 3. 需求假设

基于以上调研，形成以下产品假设：

| 编号 | 假设                                                     | 验证方式                             | 当前状态                                                                                                                |
| ---- | -------------------------------------------------------- | ------------------------------------ | ----------------------------------------------------------------------------------------------------------------------- |
| H1   | 用户愿意用邮件接收每日 AI/科技/财经精选摘要              | 种子用户订阅率 + 邮件打开率          | ✅ **已验证**：10 位种子用户中获得 8 位用户深度反馈，部分用户已形成每日查阅习惯                                         |
| H2   | Top 5 精选摘要（而非全量信息）能满足大部分用户的日常需求 | 用户满意度反馈 + 退订率              | ✅ **已验证**：用户普遍认可"高信噪比"价值，主流阅读时长 1-2 分钟即可完成，大部分用户满足于简报内容的闭环阅读            |
| H3   | 对话式交互（即时查询）能补充定时推送无法覆盖的临时需求   | 即时查询使用频率                     | ⚠️ **部分验证**：用户对 Bot 的感知度不足（浮窗引导失败），导致即时查询使用率偏低，需优化引导后重新观测                  |
| H4   | AI 自动生成的摘要质量能达到用户可接受的水平              | 用户反馈（like/dislike）+ badcase 率 | ✅ **已验证**：用户反馈内容"更加中立和事实"、"信息极度集中且有针对性"，深度用户每天节省 10-15 分钟信息获取时间          |
| H5   | 用户更信任带原文链接的 AI 摘要（vs 无来源的 AI 回答）    | 用户访谈 + 链接点击率                | ✅ **已验证**：用户主要满足于摘要闭环阅读，仅"感兴趣的内容才点击原文链接"，说明摘要本身已获信任；50% 用户愿意向他人推荐 |

### PMF 信号清单

种子测试阶段，以下信号出现即说明产品方向正确：

| 信号         | 判断标准                                                       | 说明             |
| ------------ | -------------------------------------------------------------- | ---------------- |
| 信任转移     | 用户开始直接引用日报摘要发给同事，不再每次点进原文核实         | 信任已建立       |
| FOMO 缓解    | 用户在周末敢于不打开产品，相信"如果有大事，明天的日报会告诉我" | 心理安全感建立   |
| 效率感知     | 用户认可"在这里读 5 分钟 ≈ 自己刷 30 分钟"                     | 核心价值被感知   |
| 替代方案减少 | 用户主动减少了手动刷 RSS / 社交媒体的时间                      | 产品成为信息入口 |
| 自发传播     | 用户在社群中主动推荐产品                                       | 口碑增长启动     |

## 4. 调研结论与产品方向

### 4.1 核心结论

1. **痛点真实存在**：信息过载是普遍问题，Reddit 和知乎上的讨论高度一致
2. **市场有空间**：中文市场缺少"AI 聚合 + 邮件推送 + 对话交互"的产品
3. **双模式是关键**：定时推送 + 即时查询，覆盖两种用户行为模式（JTBD 1 + JTBD 3）
4. **降噪是核心价值**：用户要的不是更多信息，而是"更少但更精准的信息"
5. **信任需要建设**：带原文链接、信源透明是建立用户信任的基础（回应 JTBD 2 和 §2.6 的担忧）

### 4.2 产品方向决策

基于调研结论，确定了 NewsFlow.ai 的产品方向：

- **形态**：Web 页面 + AI 对话浮窗 + 邮件推送（轻量，不做 App）
- **核心功能**：每日定时推送（Top 5 摘要）+ 即时新闻查询（对话式）
- **主题**：AI / 科技 / 财经（从最熟悉的领域开始）
- **差异化**：对话式交互 × 主动推送 × 中文原生
- **信任机制**：每条新闻带原文链接，信源可追溯

### 4.3 调研发现与 MVP 设计的对应关系

| 调研发现     | MVP 中的设计回应                   |
| ------------ | ---------------------------------- |
| 重复信息轰炸 | 多源聚合 + 语义去重 + Top-K 筛选   |
| FOMO 焦虑    | 每日 08:00 定时推送，提供"完成感"  |
| AI 幻觉担忧  | 每条新闻带原文链接，可回溯验证     |
| 工具碎片化   | 一个产品覆盖"查+订+收"三条链路     |
| 信息茧房担忧 | 不做个性化推荐，信源透明可控       |
| 中文市场空白 | 中文原生，支持中英文输出           |
| 开箱即用需求 | 首页输入邮箱即可订阅，无需复杂配置 |

### 4.4 调研局限性（诚实说明）

- **样本量小**：非正式沟通对象有限，主要是身边的科技圈朋友，存在选择偏差
- **缺少定量数据**：没有做问卷调查或大规模用户访谈
- **自身偏见**：作为产品的创造者和用户，可能高估了痛点的普遍性
- **社区数据的代表性**：Reddit 以英文技术用户为主，知乎以中文知识工作者为主，两者都不能完全代表目标用户群体
- **引用的局限**：社区调研通过 Perplexity 搜索聚合，部分引用为 AI 整理后的摘要而非逐条手动摘录的原帖链接

这些局限性将通过后续的种子用户测试来弥补。

## 5. 后续验证计划

### 5.1 种子用户测试（✅ 已完成，2026-02-28）

> 详细报告见：[`seed_user_research_report.md`](../../seed_user_research_report.md)

| 验证项       | 方法                         | 成功标准        | 实际结果                                                    |
| ------------ | ---------------------------- | --------------- | ----------------------------------------------------------- |
| 订阅意愿     | 邀请 10 人试用，观察订阅转化 | 转化率 > 50%    | ✅ 10 人全部完成订阅，7 人提交深度反馈（70%）               |
| 内容满意度   | 1v1 沟通 + like/dislike 数据 | 满意度 > 70%    | ✅ 用户普遍认可核心价值，认为内容"中立、集中、有针对性"     |
| 留存         | 观察 7 天后是否仍在使用      | 7 日留存 > 60%  | ✅ 部分用户已形成每日查阅习惯（通勤场景），50% 用户愿意推荐 |
| 即时查询价值 | 观察使用频率和场景           | 周均使用 > 2 次 | ⚠️ 浮窗 Bot 引导失效，用户未注意到可交互，使用率偏低        |
| 退订原因     | 退订时收集原因               | 收集定性反馈    | ✅ 无退订；主要摩擦点为推送时间一刀切（早 8 点）            |
