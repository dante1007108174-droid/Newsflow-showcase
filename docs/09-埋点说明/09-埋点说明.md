# 埋点（Analytics）使用说明

本文件描述 Daily AI News 的埋点 MVP：数据结构、上报方式、约束与调试方法。

## 1. 先决条件：建表 SQL

请先在 Supabase 的 SQL Editor 执行：`docs/analytics_events_table.sql`。

## 2. 数据模型（events 表）

所有埋点统一写入 Supabase `public.events`：

- `id`：UUID，默认 `gen_random_uuid()`
- `user_id`：TEXT，前端 localStorage 持久化用户标识
- `session_id`：TEXT，前端 sessionStorage 会话标识
- `event_name`：TEXT，事件名
- `event_data`：JSONB，事件元数据（禁止写入敏感字段）
- `created_at`：TIMESTAMPTZ，服务端写入时间

## 3. 事件清单（MVP）

说明：你选择了“策略 1（只埋后端链路）”，所以与 WebChat SDK 对话链路强相关的 `message_sent` / `bot_reply` 作为 post-MVP。

前端通过 `/api/track` 上报（白名单）：
- `page_view`
- `client_error`

后端自动写入（不走前端上报）：
- `workflow_call`（`POST /api/workflow/send-test` 成功/失败都会写）
- `email_sent`（同上，基于当前链路的“触发成功/失败”信号）
- `subscribe`（`POST /api/subscription/upsert` 成功写）
- `feedback`（`POST /api/feedback` 成功写）

## 4. 前端如何上报

实现位置：`frontend/js/app.js`

前端会自动：
- 生成稳定 `user_id`（localStorage key: `daily_ai_news_user_id`）
- 生成 `session_id`（sessionStorage key: `daily_ai_news_session_id`）
- 在 `DOMContentLoaded` 触发 `page_view`
- 监听 `window.error` / `unhandledrejection` 触发 `client_error`

上报接口：
- `POST /api/track`

请求头（推荐同时带上，便于后端统一提取身份）：
- `X-User-Id: <user_id>`
- `X-Session-Id: <session_id>`

请求体示例：

```json
{
  "event_name": "page_view",
  "event_data": {
    "url": "/",
    "referrer": "",
    "ua": "Mozilla/5.0 ..."
  }
}
```

## 5. 后端如何写入（内部事件）

核心实现：`backend/lib/tracking.js`

后端路由中通过：
- `extractIdentity(req)`：从 body 或 headers 读取 `user_id/session_id`
- `trackServerEvent({ eventName, userId, sessionId, eventData })`：写入 Supabase

重要约束：
- 埋点是 best-effort：写失败不会影响主接口返回（失败时一般降级为 202 / warn log）
- `event_data` 限制：必须是对象，且 JSON 序列化后不超过 4096 bytes
- 敏感字段拦截：`event_data` 中禁止出现这些 key（大小写不敏感）：
  - `email`
  - `message_content`
  - `raw_message`
  - `token` / `authorization`
  - `password` / `secret` / `api_key`

## 6. /api/track 行为说明

实现位置：`backend/routes/track.js`

- 仅允许前端事件白名单：`page_view`、`message_sent`、`client_error`（MVP 实际用到前两者中的 `page_view` + `client_error`）
- 限流：60 次/分钟/客户端（按 IP），超过返回 429
- 当 Supabase 未配置或写入失败：返回 202（不会让前端报错阻塞体验）

## 7. 快速自测（curl）

1) 发送一个正常 page_view：

```bash
curl -i -s -X POST "http://localhost:3000/api/track" \
  -H "Content-Type: application/json" \
  -H "X-User-Id: smoke_user" \
  -H "X-Session-Id: smoke_session" \
  -d '{"event_name":"page_view","event_data":{"url":"/","referrer":""}}'
```

预期：
- 如果 Supabase 已建表且环境变量正确：200 + `{"success":true}`
- 如果 Supabase 未就绪：202（降级）

2) 验证敏感字段拦截：

```bash
curl -i -s -X POST "http://localhost:3000/api/track" \
  -H "Content-Type: application/json" \
  -d '{"event_name":"page_view","event_data":{"email":"leak@test.com"}}'
```

预期：400 + `event_data 包含敏感字段`
