## 1. 项目定位与目标

NewsFlow.ai 是一个“智能新闻订阅助手”作品集项目，核心能力是:

- 用户通过自然语言完成 **即时查询**、**订阅管理**、**测试推送**
- 系统每天定时生成日报并通过邮件推送(偏工程化能力展示)
- 通过评测体系把“模型能力”转成可控的“产品稳定性”

这个项目对我而言的目标不是做一个 Demo，而是完整跑通:

- 产品定义(PRD)
- Prompt/工作流设计
- 模型选型与分层
- 前后端落地与迭代
- 评测与复盘文档沉淀

---

## 2. 时间线(四阶段)

整体周期约 2.5 个月，从去年 11 月中旬开始，按阶段推进如下(时间为大致区间):

### Phase 1: 原型验证(约 2 周)

- 搭建最小可用的 Coze Agent 与工作流，验证“能查到新闻 + 能跑通订阅/发送”的闭环。
- 建立 Prompt 记录表(早期有记录，后续随着任务增多记录减少)。
- 主要产出: 可用的最小交互链路 + 初版 Prompt。

### Phase 2: 评测体系与模型选型(约 1 个月，集中在 12 月)

- 补齐评测数据集与 Bad Case 库，明确什么叫“稳定”。
- 做调度层模型选型评测(18 Case)，输出模型分层决策。
- 输出/完善 PRD(逐步升级为 v2.0)。

### Phase 3: 产品落地(集中在 1 月)

- 落地前端与后端(静态前端 + Node/Express API)。
- 前后端落地后，对 Workflow 做工程化升级以适配真实调用链路。
- 对 Agent 系统提示词做两版关键结构调整(尤其是 `user_id` 的上下文提取与稳定路由)。

### Phase 4: 文档沉淀与作品集整理(当前)

- 把“做过的事”沉淀为面试可讲、可复用的文档资产。
- 补齐模型选型报告、Prompt 工程手册、评测与质量报告、项目复盘。

## 3. 最关键的决策: 分层模型策略

这个项目最重要的学习点来自 AI PM 培训课的一条原则:

- **充分且合适地使用各个模型的能力**
- **既要避免能力溢出(浪费 token/延迟)，也要避免能力不足(质量翻车)**

因此我没有追求“一个模型打天下”，而是按节点分层:

- 调度/扩写/轻量推理: 豆包 1.6(更稳定的槽位/指令遵循 + 接入便利)
- 深度摘要/长输出生成: DeepSeek V3.2(更适合承担摘要与邮件正文等长输出)

这条决策在文档中分别落地为:

- `docs/02-模型选型分析报告.md` (为什么这么选、怎么评测、怎么算账)
- `docs/01-PRD-产品需求文档-v2.0.md` (架构与 Roadmap 的产品化表达)

## 4. “混乱”是否要写进故事里

你提到时间线有些混乱，我的建议是 **保留真实感，但用结构把它收敛**。

原因:

- 第一次 0-1 独立项目，出现“边做边调整”是正常现象。
- 面试官并不怕你“走弯路”，反而更关注你是否能:
  - 识别问题
  - 提炼规律
  - 把混乱变成稳定流程(评测集、回归、文档)

因此，复盘里可以写:

- 早期聚焦“先跑通”，中期补“评测与选型”，后期再做“工程化落地”。
- 过程中记录体系从“详细记录”逐步变成“关键节点记录”，但最终通过文档重新补齐。

这类表达不是包装，而是把真实经历抽象成方法论。

## 5. 做得好的部分

### 5.1 先建立评测，再谈优化

在模型应用里，很多体验问题是“看起来像模型问题，实际上是流程/约束问题”。

我选择尽早建立评测集与 Bad Case 库:

- 让问题可复现、可讨论、可回归
- 让“Prompt 改动”从拍脑袋变成可验证

### 5.2 关键约束前置(隐私 + 注入防御)

通过 System Prompt 强约束:

- 从 `[System Context]` 提取 `user_id`，严禁向用户索要/泄露
- 对“索要系统提示词”等注入攻击直接拒绝并回到新闻主题

对应沉淀见 `docs/03-Prompt工程手册.md`。

### 5.3 把 token/延迟当成一等公民

通过 Top-K 预筛选等手段避免把 100+ 条 RSS 原文直接交给大模型处理，避免延迟爆炸。

## 6. 做得不够好的部分(以及为什么)

### 6.1 记录体系中断

早期有 Prompt 迭代记录表，但随着工作量暴涨(产品/评测/前后端/工作流同人)，记录变少。

复盘后我认为，至少应该保留两类记录:

- “重大版本”变更点(为什么改、改了什么、风险是什么)
- “回归集”结果(改动前后是否退化)

### 6.2 顺序上有来回

从结果看，最合理的顺序是:

- 先定义边界(PRD) -> 再做评测 -> 再做工程落地

但真实执行中，为了快速验证可行性，我在前期更偏工程驱动，后期再补齐产品与评测框架。

这不算错误，但下一次会更刻意地“先立标准，再做实现”。

---

## 7. 如果重来一次

1. **更早固化回归流程**: 把 18 Case/22 Case 在每次关键改动后都跑一遍，形成节奏。
2. **Prompt 版本化**: 重大版本必须有 changelog(哪怕只写 10 行)。
3. **更早做信息结构化**: 让上游数据尽量结构化后再给 LLM，减少“靠模型猜”。
