## 0. 概览

系统中共有 **4 个代码节点**，全部用 Python 编写，运行在 Coze Workflow 的代码节点环境中。

| # | 代码节点 | 所属流程 | 核心职责 | 代码行数 |
|---|---------|---------|---------|---------|
| 1 | AI和科技主题RSS解析 | 邮件推送 | 抓取12个RSS源，过滤/去重/均衡后输出新闻列表 | 545行 |
| 2 | 财经主题RSS解析 | 邮件推送 | 抓取7个财经RSS源，并发+重试，输出新闻列表 | 564行 |
| 3 | HTML渲染 | 邮件推送 | 接收LLM编辑结果，生成邮件HTML | 249行 |
| 4 | 即时查询RSS解析 | 即时查询 | 抓取8个通用RSS源，并发抓取，输出新闻列表 | 347行 |

### 数据流位置

```
[邮件推送流程]
关键词扩写(LLM) → RSS解析(代码节点1/2) → 新闻编辑(LLM) → HTML渲染(代码节点3) → 发送邮件

[即时查询流程]
关键词扩写(LLM) → RSS解析(代码节点4) → 新闻摘要(LLM) → 返回给用户
```

代码节点在流程中的角色：**承上启下的数据加工层**。上游是 LLM 扩写的关键词，下游是 LLM 做新闻编辑/摘要。代码节点负责"脏活"——网络抓取、XML解析、去重、均衡——这些是确定性逻辑，不适合交给 LLM。

## 1. RSS解析代码节点（共3个）

### 1.1 共同的技术模式

三个 RSS 解析节点共享大量相同的底层逻辑，区别主要在 RSS 源列表、抓取策略和输出细节。

#### 入口签名

```python
async def main(args):
    # Coze 代码节点统一入口
    # args 是一个字典，包含上游节点传入的参数
```

#### 输入参数

```python
# 从 args 中获取搜索关键词，兼容两种传参方式：
raw_input = args.get('params', {}).get('search_keywords') or args.get('search_keywords')
```

- `search_keywords`：上游"关键词扩写"LLM 节点输出的扩展词列表
- 支持格式：JSON数组字符串 `["AI","人工智能"]`、逗号分隔字符串 `"AI,人工智能"`、直接list

#### 输出结构

```python
return {
    "result": result_list,      # 新闻列表，每条含 title/description/link/time_code/source
    "debug_info": debug_info,   # 文本格式的抓取报告（信源明细、过滤漏斗）
    "current_time": "2026-02-28 08:00"  # 北京时间
} 
```

#### 共享工具函数

| 函数 | 作用 | 关键逻辑 |
|------|------|---------|
| `get_beijing_time()` | 获取北京时间 | UTC+8，不依赖系统时区 |
| `is_chinese_content(text)` | 判断是否中文内容 | 中文字符占比 ≥ 30% 才保留 |
| `process_content(html)` | 清洗HTML为纯文本 | 去标签、去换行、截断350字 |
| `parse_time(raw_time)` | 解析各种时间格式 | 依次尝试 RFC2822 → 自定义 → ISO8601，全部失败则用当前时间 |
| `extract_keywords(text)` | 提取关键词集合 | 英文品牌词 + 中文2-4字词 + 数字，去停用词 |
| `extract_trending_words(items)` | 提取热点词 | 统计标题中高频词，出现≥2次才入选 |

#### 处理管线（Pipeline）

所有 RSS 解析节点都遵循相同的处理管线：

```
RSS抓取 → 中文过滤 → 24h时效过滤 → 关键词匹配 → 语义去重 → 信源均衡 → 排序 → 输出
```

每一步都会统计数量，最终输出 debug_info 中的"过滤漏斗"：

```
原始抓取(中文): 180 条
24h时间过滤后: 120 条
关键词匹配后: 85 条
语义去重后: 52 条
信源均衡后(cap80): 48 条
最终输出: 48 条
```

### 1.2 语义去重算法（核心算法，三个节点共用）

这是代码节点中最复杂的部分，目的是把多个信源报道同一事件的重复新闻合并。

#### 相似度计算 `calculate_semantic_similarity()`

```
基础分 = Jaccard相似度(标题关键词 ∪ 描述前100字关键词)

如果共享核心实体（OpenAI/Google/华为/腾讯等30+个）→ +0.3
如果共享核心实体 + 同类事件动作（融资/发布/合作）→ 再+0.2

最终分 = min(1.0, 基础分 + 加成)
```

- 核心实体表：`openai, google, microsoft, apple, nvidia, meta, amazon, anthropic, deepseek, 字节, 华为, 腾讯, 阿里, 百度, 小米, 特斯拉, tesla, spacex, 英伟达, minimax, samsung, 比亚迪, 蔚来, 理想, 小鹏`
- 事件动作模式：融资类（融资/注资/投资/估值）、发布类（发布/推出/上线/发售）、合作类（合作/联合/携手）

#### 去重策略 `smart_deduplicate()`

- 阈值：**0.50**（相似度超过50%判定为重复）
- 排序优先：描述长度长的排前面（内容更丰富的优先保留）
- 合并时记录：`_heat_count`（报道热度）、`_sources_list`（来源列表）、`_best_link`（最佳链接，取内容最丰富版本的链接）

#### 信源均衡 `balance_sources()`

防止某个信源霸占输出：
- 总上限：80条
- 单源上限：总数 × 40%（至少3条）
- 分配策略：先每源保底3条，再轮询补充

## 2. HTML渲染代码节点

**文件**：`html渲染代码节点.md`

### 2.1 职责

接收上游 LLM 新闻编辑节点的输出（JSON格式的 top_news + quick_news），渲染成可直接发送的邮件 HTML。

### 2.2 输入

```python
llm_result_ai = args.params.get('llm_result_ai')       # AI主题LLM结果
llm_result_tech = args.params.get('llm_result_tech')     # 科技主题LLM结果
llm_result_finance = args.params.get('llm_result_finance') # 财经主题LLM结果
keyword = args.params.get('keyword', '热点')              # 当前主题
```

三个 LLM 结果对应三条分支路由，实际只有一个非空。代码会依次尝试解析，取第一个有效的。

### 2.3 输出

```python
return {"final_html": html_content}  # 完整的HTML邮件字符串
```

### 2.4 核心逻辑

#### LLM结果解析 `parse_llm_result()`

兼容多种 LLM 输出格式：
- 直接是 dict 且含 `top_news`/`quick_news` → 直接用
- dict 里有 `output` 字段（字符串或嵌套dict）→ 解包
- 纯字符串（可能带 ```json 标记）→ 清洗后 JSON.parse
- 全部失败 → 返回错误HTML（红色提示框 + debug信息）

#### 智能摘要截断 `smart_truncate_summary()`

目标：在120字附近找到自然的句子断点，避免截断在词语中间。

```
优先级：
1. 120-180字范围内找句号/问号/感叹号 → 在此处截断
2. 找分号/冒号 → 在此处截断
3. 向前回退找上一个完整句子（至少保留78字）
4. 找逗号 → 截断并加"……"
5. 都没有 → 硬截断180字加"……"
```

## 3. 技术债与已知问题

### 3.1 代码重复

三个 RSS 解析节点有大量重复代码（工具函数、去重算法、均衡算法几乎一样）。Coze 代码节点不支持跨节点共享模块，所以只能复制粘贴。

**风险**：修改去重算法时需要同步改3个地方，容易遗漏。

### 3.2 AI/科技版仍是串行抓取

和即时查询版都已升级为并发抓取，但 AI/科技版仍然是串行的。12个源串行抓取，最坏情况下需要 12×15=180秒。

### 3.3 即时查询版没有去重

即时查询版把去重完全交给了下游 LLM。当多个源报道同一事件时，LLM 会收到大量重复内容，浪费 token 且可能影响摘要质量。

### 3.4 RSS源URL的脆弱性

- 部分源使用 `rsshub.rssforever.com` 作为中转（第三方 RSSHub 实例），该服务不保证可用性
- 36氪在推送版用 `rss.aishort.top/?type=36kr`，在即时查询版用 `36kr.com/feed`，URL不一致
- 中新网和新华社使用 HTTP（非HTTPS），部分环境可能拦截

### 3.5 参数获取方式不统一

- AI/科技版：有复杂的防御性参数获取（多层try/except）
- 财经版和即时查询版：简洁的单行获取
- 说明 AI/科技版可能遇到过参数传递问题，打了补丁

## 4. 节点间数据流对照

### 邮件推送流程

```
[关键词扩写LLM]
    输入: keyword (如 "AI")
    输出: search_keywords (如 ["AI","人工智能","大模型","GPT",...])
        │
        ▼
[RSS解析代码节点] ← 节点①或②，按主题路由
    输入: search_keywords
    输出: result(新闻列表) + debug_info + current_time
        │
        ▼
[新闻编辑LLM] ← AI/科技/财经各有独立提示词
    输入: news_list=result, current_time, keyword
    输出: {top_news: [...], quick_news: [...]}
        │
        ▼
[HTML渲染代码节点] ← 节点③
    输入: llm_result_ai/tech/finance, keyword
    输出: final_html
        │
        ▼
[发送邮件]
```

### 即时查询流程

```
[关键词扩写LLM]
    输入: keyword
    输出: search_keywords
        │
        ▼
[RSS解析代码节点] ← 节点④
    输入: search_keywords
    输出: result(新闻列表) + debug_info + current_time
        │
        ▼
[新闻摘要LLM]
    输入: 新闻列表 + raw_query + keyword + current_time
    输出: 格式化的新闻摘要文本（直接返回给用户）
```
