# LLM Agent 评测闭环（LLM Agent Evaluation Loop）

用于 router/dispatcher/workflow 的质量工作：必须用可复现的证据证明“真的修好了”，而不是靠主观感觉。

## ANALYSIS MODE（分析模式：先收集上下文再深入）

在动手改动之前，先并行 gather context，避免方向跑偏。

CONTEXT GATHERING（并行）：
- 1-2 个 explore agents（代码库模式、现有实现）
- 1-2 个 librarian agents（若涉及外部库/平台行为）
- 直连工具：Grep / AST-grep / LSP 做定向搜索

如果复杂 —— 不要硬扛，找专家：
- Oracle：常规复杂问题（架构、调试、复杂逻辑）
- Artistry：非常规问题（需要另辟蹊径）

先综合结论再推进。

## 触发语句
- “设计一套评测集 / evaluation set”
- “跑 baseline 并分析 badcase”
- “为什么这个 dispatch case 不稳定”
- “修失败用例并回归”
- “做多轮稳定性验证”

## 必要输入
- Suite 路径（例如 `tests/dispatch_suite.json` 或 `tests/agent_e2e_suite.json`）
- Runner 命令（例如 `python tests/run_dispatch_suite.py` / `python tests/run_agent_e2e_suite.py`）
- 用例级 PASS/FAIL 的提取规则（字段名、判定口径）
- 产物目录（report JSON/MD/XLSX 输出位置）
- 跑测约束（token/配额、最大调用数、重跑策略、是否允许全量）

## 工作原则（Operating Rules）
- 先假设后验证：每一次重跑必须验证一个明确的改动假设。
- 修复尽量小：修 bug 时不做无关重构。
- 先分清 `infra`（超时/网络/平台抖动）与 `logic`（路由/决策）再下结论。
- 单次迭代只改一个轴：优先“只改 prompt”或“只改 runner/解析”。
- 每个阶段都保留证据：报告文件路径、统计指标、失败列表、关键日志。

## 13 步工作流

1. **定义评测契约（Evaluation Contract）**
   - 定义维度（意图、参数补槽、连续性、安全、格式等）。
   - 每个维度写清楚 PASS/FAIL 量表与判定口径。
   - 定义稳定性门槛（例如“5 轮 >= 90%”）。

2. **设计测试集（Test Set）**
   - 覆盖：主路径、歧义表达、对抗语句、多轮连续性。
   - 为每条用例打标签：意图、必填槽位、预期行为。

3. **覆盖审计（Coverage Audit）**
   - 跑之前对照需求矩阵检查覆盖。
   - 语言分布/意图分布要“刻意设计”，不要“偶然凑出来”。

4. **基线跑测（Baseline Run）**
   - 全量跑 1 次。
   - 记录：通过率、失败数、失败用例 ID。
   - 保存带时间戳的产物（JSON/MD/XLSX）。

5. **评估器校准（Evaluator Calibration）**
   - 抽查 10-20% 的 PASS/FAIL 标签。
   - 先确认 parser/judge 没问题再相信指标。
   - 如果评估器错了：先修评估器，再重跑 baseline。

6. **Badcase 分诊（Triage）**
   - 每个失败先分类：
     - `logic`：路由/决策错误
     - `format`：输出契约/解析不匹配
     - `infra`：超时/网络/平台瞬态异常
   - 只有 `logic` 与 `format` 进入“行为修复”；`infra` 走稳定性与隔离策略。

7. **根因定位（Prompt/Script/API）**
   - 每个失败归因到主因：
     - prompt 指令缺口/冲突
     - runner/解析 bug（会话处理、参数提取、流式拼接）
     - API/平台行为（审核延迟、缓存、记忆）
   - 对每个簇写一句“期望行为”。

8. **变更影响分析（Change Impact）**
   - 预测哪些原 PASS 用例可能回归。
   - 构建“受影响小集合”，用于快速回归。

9. **分层修复（Layered Fixes）**
   - 分层做小步改动：
     - A：prompt 约束/优先级
     - B：few-shot 反例/对抗样例
     - C：解析器/后处理护栏
   - 避免“大改 prompt + 大改代码”同一补丁。

10. **定点重跑（Targeted Reruns）**
   - 只跑失败/受影响用例（对不稳定用例默认 5 轮）。
   - 只有同时满足：指标过门槛 + 行为符合量表，才接受。

11. **全量回归（Full Regression）**
   - 定点通过后再跑全量。
   - 对比 baseline vs 当前：提升幅度、新增坏例、修复保留率。

12. **多轮稳定性验证（Stability）**
   - 对高风险用例（或全量）做多轮重复。
   - 低于门槛则回到第 7 步，更新假设继续迭代。

13. **反馈闭环（Suite Feedback）**
   - 将已解决 badcase 固化进长期评测集。
   - prompt/model/runner 变更触发回归。
   - 维护一套小型 smoke set 做快速信心验证。

## 护栏（Guardrails）
- 没有新假设，不允许盲目全量重跑。
- `infra` 失败不要当成行为回归。
- 没有“定点多轮证据”，不要宣称修复完成。
- 如果用户策略禁止自动重跑：先停下确认再继续。

## 通用测试集模板（Suite JSON：平台无关）

目标：把“评测契约/维度/门槛/用例结构”声明在数据里，让 runner 尽可能少改代码。

### 顶层结构（推荐固定 5 块）

- `测试集`：元信息（名称、版本、说明、目标平台/模型可选）
- `默认配置`：运行默认参数（user_id、邮箱、流式开关、超时、配额约束等）
- `评测维度`：D1-D5（或你自定义的维度）声明式定义：权重/判定方式/通过标准
- `验收门槛`：整体与各维度的门槛（通过率/平均分/稳定性等）
- `用例`：用例数组（单轮/多轮，预期结果与评测维度开关）

### 用例结构（推荐字段约定）

每条用例建议包含：
- 用例信息：`用例ID`、`意图分类`、`二级意图`、`测试场景`
- 对话信息：`对话类型`（单轮/多轮）、`会话组ID`（多轮必填）、`轮次`、`用户输入`
- 预期结果：`预期结果`（预期行为/技能/动作/参数 + 回复约束）
- 维度开关：`评测维度`（这一条用例纳入哪些维度）
- 运行填充：`输出结果`、`是否通过`、`失败原因`、`修复状态`、`备注`

### 多轮用例规则（强约束）

- 同一会话组内：`会话组ID` 必须一致，`轮次` 必须连续且可排序。
- 多轮跑测如果被分片执行：必须确保同一 `会话组ID` 仍复用同一个会话（conversation/session）。
- 对“缺参追问”类用例：推荐在 runner 支持 `context_user_id` 覆盖，确保使用 fresh `user_id`，避免历史记忆干扰评估结论。

## 通用报告模板（Excel：5 Sheet 架构与表头）

目标：报告结构固定、字段可比、便于跨轮次/跨项目对比。

### Sheet 1：测试汇总（推荐两块结构）

块 1：运行概览（两列：指标 / 值）——建议固定这些行：
- 执行时间
- Run ID
- Bot ID（或 Agent/Model ID）
- 流式首响（TRUE/FALSE）
- 总用例 / 通过 / 失败 / 通过率
- D1_意图路由（通过率）
- D2_参数完整（通过率）
- D3_工具/工作流执行（通过率）
- D4_回复质量_平均分
- D5_交互规范（通过率）
- D5_交互规范_平均分
- 首字响应ms：{P50, P90, 均值}
- 总耗时ms：{P50, P90, 均值}
- 单轮通过率
- 多轮通过率

块 2：验收指标（四列：验收指标 / 实际值 / 门槛 / 状态）
- 状态建议输出：`达标` / `未达标`

口径建议：
- 通过率类用例：按“该维度纳入评测”的用例分母计算（如果没有维度开关，默认用总用例数）。
- 平均分：只对有分数的用例求均值（过滤空值）。
- P50/P90：以 ms 数组计算分位数（线性插值或 nearest 都可，但要写清口径并保持一致）。

### Sheet 2：测试明细（每行 1 个用例）

推荐表头（可按需要加列，但不要随意改名）：

`用例ID, 意图分类, 二级意图, 测试场景, 对话类型, 会话组ID, 轮次, 用户输入,
 预期行为, 预期技能, 预期动作, 预期参数,
 实际行为, 实际技能, 实际参数, tool_output摘要,
 首字响应ms, 总耗时ms,
 D1通过, D2通过, D3通过, D4评分, D5评分,
 是否通过, 失败原因, 修复状态, 备注`

说明：
- “摘要”列用于减少 Excel 体积；完整原文可放入 JSON 报告或单独附件。
- 对多轮：每轮一行，便于定位哪一轮失败。

### Sheet 3：验收标准 / 评判标准

建议包含 3 张小表：

表 A：维度定义与权重
- 维度 / 权重 / 评判方式 / 通过标准 / 说明

表 B：评分量表（D4/D5 如使用 1-5 分）
- 评分 / 标准 / 示例

表 C：用例通过判定规则（按场景类型）
- 场景类型 / 通过条件 / 说明

### Sheet 4：失败案例分析

推荐表头：

`用例ID, 测试场景, 对话类型, 预期技能, 实际技能, D1/D2/D3, D4/D5,
 失败类型, 失败原因, 诊断, 建议, 修复状态`

失败类型建议枚举：`意图路由错误` / `参数提取错误` / `工具执行失败` / `回复质量不达标` / `交互规范违规` / `infra`。

### Sheet 5：迭代追踪（跨轮次对比）

推荐表头：

`用例ID, 测试场景, 基线, 第1轮修复, 第2轮回归, 当前状态, 变更记录`

当前状态建议枚举：
- `✅稳定`：一直 PASS
- `✅已解决`：曾 FAIL，修复后连续 PASS
- `⚠️回归`：曾 PASS，最近 FAIL
- `⚠️待稳定`：修复后只绿过 1 次，未验证稳定性
- `❌未解决`：仍 FAIL


## Coze / Agent 评测适配（强经验项）

### 多轮连续性
- 明确保持同一会话：同一个 `会话组ID` 对应同一个 conversation。

### infra 分类建议
- 流式传输异常（例如 `ChunkedEncodingError`）按 `infra` 处理。
- 工作流偶发报错/超时优先归到 `infra`，不要立刻改 prompt。

### “记忆干扰”隔离
- 评测“缺参追问”的用例必须使用全新的随机 `user_id`。
- 否则数据库/记忆中已有邮箱等信息会导致 Agent 自动补全，跳过追问（假阴性）。

### STUB + function_call 参数合并
- STUB 模式下结构化参数可能分散在 `stub_calls` 与 `function_calls` 中。
- 需要做 deep merge 才能得到完整参数（避免类似 `raw_query` 丢字段）。

### 发布/审核延迟
- 发布后的 bot 可能存在审核/生效延迟：控制台预览与 API 结果不一致时，先排查是否仍在审核。

## 证据清单（Evidence Checklist）
- Baseline 产物路径
- 定点重跑产物路径（可多份）
- 全量回归产物路径
- 稳定性验证产物路径
- Before/After 指标（`x/y`、`%`、delta）
- 每个修复用例的根因记录（1-3 句即可）

## 报告模板（Markdown）

```markdown
## 评测结果
- 契约：<量表 + 门槛>
- Baseline：<x/y, %>
- 当前：<x/y, %>
- 提升：<+n cases, +m%>

## 已修复用例
| 用例 | 分类 | 根因 | 修复 | 稳定性 |
|---|---|---|---|---|
| D016 | logic | 槽位优先级漂移 | prompt 锚定邮箱优先 | 5/5 |

- 修复保留：<列表>

- <infra 抖动或未澄清的歧义>

## 证据
- Baseline：<path>
- Targeted：<path>
- Regression：<path>
- Stability：<path>
```

## 反模式（Anti-Patterns）
- 没有明确量表/门槛就开始设计用例
- 未校准评估器就相信指标
- 一次补丁混合大量 prompt 改动和代码改动
- 只跑 1 次绿就收工，不做稳定性验证
